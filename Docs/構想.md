# ゲーム構想（青写真）

最終更新: 2026-03-05

## 0. 一言ピッチ

- 中世ファンタジー世界で冒険者ギルドを経営し、依頼と冒険者を見極めて最適に仲介する「斡旋シミュレーション」。

## 1. プレイヤー体験の核

- どんな感情を主に味わってほしいか: 目利き、采配、経営判断の責任感
- どんな達成感を提供するか: 難しい依頼を適切な冒険者へつなぎ、損害なく成功させる達成感
- 1プレイ（または1セッション）の理想時間: 20〜30分
- 優先度: 人材采配 > 経営快感 > 世界への影響（影響要素は低優先）

## 2. コアループ（仮）

1. 日中に依頼を受ける（持ち込み / 営業で取りに行く / 急な割り込み）
2. 依頼情報を精査し、掲示・保留・限定公開を判断する
3. 冒険者と個別面談して割り当てを決める（質問対応、編成、事前助言）
4. 冒険者を出発させる
5. `次の日` ボタンで時間を進める
6. 帰還した冒険者から結果報告を受け、評判と個人履歴を更新する
7. 次の依頼導線を作る（依頼主フォロー、冒険者ケア、営業）

### 仮説

- 「情報の不完全性（自己申告）」が判断ゲームとしての核になる
- 依頼の取捨選択より、公開方法と説明責任がギルド運営らしさを作る
- 冒険者ごとの性格・能力・相性の同時最適化が面白さを作る

### 決定

- プレイヤーは冒険者本人ではなく「ギルド側の仲介者」
- 世界観は中世ファンタジーをベースにする

### 保留

- 死亡/重傷時の離脱期間レンジと退団確率
- 依頼ボードUIの可読性（件数増加時の閲覧ストレス）

### メインループ詳細（v0.2）

#### ループ前提

- 固定フェーズ進行は採用しない（日中はプレイヤーが自由に運用する）
- 時間は `次の日` ボタンで進む
- 出発済みの冒険者は、経過日数に応じて帰還し結果報告を行う
- 情報は常に「申告情報」として扱い、真実は直接表示しない

#### 1日の運用サイクル

1. 依頼対応（受理 / 保留 / 取り下げ対応 / 営業獲得）
2. 依頼掲示の調整（公開範囲、メモ、優先度）
3. 冒険者との個別面談（希望確認、質問応答、提案）
4. 割当確定と出発前助言（装備/方針）
5. `次の日` を押して日付を進める
6. 帰還者の結果報告を受ける（要約3行 + 詳細ログ）
7. 評判・履歴更新と次の対応準備

#### ループの面白さを作る要点

- 日中にやることの順序を固定せず、プレイヤーの裁量で運用できるようにする
- 個別面談で情報が増える構造にして、一覧操作だけの作業化を防ぐ
- 失敗を罰で終わらせず、次の判断材料（履歴/関係）へ変換する

## 3. 世界観・テーマ

- 舞台: 魔物被害と遺跡探索需要が高い交易都市と周辺領地
- 時代 / テクノロジー: 中世ファンタジー（魔法あり、産業は手工業中心）
- キーワード3つ: 信用経済 / 人材マッチング / 冒険リスク管理

## 4. プレイヤー行動と操作

- 基本アクション: 依頼獲得、掲示判断、冒険者ヒアリング、割当確定、出発前助言、結果処理
- 失敗時のペナルティ: 主に評判低下、冒険者の長期離脱、冒険者離脱（退団）リスク
- リカバリー手段: 低難度案件で信用回復、安定運用で関係値を再構築

## 5. 成長要素

- 成長対象（キャラ/拠点/装備など）: 冒険者個人の履歴と関係性、ギルド運用枠、情報管理機能
- 成長リソース: 資金、評判、依頼主ネットワーク、冒険者との信頼度
- リプレイ動機: 冒険者の喪失/復帰/成長で毎回異なる人間関係ログが生成される

## 6. 勝利条件・終了条件

- クリア条件: なし（エンドレス運営）
- ゲームオーバー条件: 原則なし（極端な信用低下時も立て直し可能なソフト破綻設計）
- 継続要素: 日々の運用ログ、冒険者個人史、依頼主との関係史を蓄積

## 7. 対象プラットフォーム

- 想定デバイス: PC（将来的にコンソール展開検討）
- 入力手段: マウス＋キーボード中心
- オフライン/オンライン: 基本オフライン（ランキング等のみ任意オンライン）

## 8. まず作る最小プロトタイプ（MVP）

- 目的: 「依頼選別 + 人材マッチング」が単体で面白いかを検証する
- 実装範囲: 1都市、依頼30〜50件プール、冒険者12人、結果自動解決、プレイヤー操作起点の時間進行
- 検証したい仮説: 最適編成を考える時間がストレスではなく快感になるか / 冒険者への愛着が生まれるか / 不完全情報でも判断が楽しいか
- 期間目安: 4〜6週間

## 9. 差別化ポイント（仲介業メタファー）

- 「強い冒険者を育てる」より「適切に案件へ割り当てる」ことを主題にする
- 依頼主・冒険者の双方満足を取る三者調整（報酬、納期、安全）
- 高報酬案件ほど契約条件が厳しく、仲介者の交渉力が問われる

## 10. 潜在要件（ヒアリング一次）

### 方向性（確定）

- 体験優先度は「人材采配の快感」>「経営の快感」>「世界への影響」
- 作業感は避ける（ただし運要素・理不尽さは世界のリアリティとして許容）
- 依頼失敗は許容し、失敗結果は数字だけでなく冒険者個人の経験として蓄積させる
- 冒険者は「愛着を持つ個人」として扱う
- 冒険者の死亡/引退は実装対象
- 戦闘は完全自動解決（プレイヤー介入は出発前準備のみ）
- 慌てさせない進行設計（プレイヤーの操作起点で進むテンポ）
- ゴールは置かず、エンドレス運営型
- ペルソナは経営シム層 + TRPG/小説嗜好層

### 重要な示唆

- 依頼を大量処理させるゲームにはしない（案件把握ストレスを抑える）
- 依頼主/冒険者とのやりとり自体を主要コンテンツにする
- 高難度案件への誤配属が、死亡リスクとギルド評価悪化につながる緊張感を作る

### 未確定（次ラウンドで決める）

- 死亡の重さ: 永久ロスト中心 / 高コスト蘇生あり
- 進行方式: ターン制 / 一時停止可能な常時進行
- 1サイクルの案件量: 少数濃密 / 中量フィルタ運用
- 人間関係ドラマの実装粒度: テキスト中心 / イベント分岐中心

## 11. 潜在要件（ヒアリング二次）

### 確定

- 冒険者の死亡/重傷は「長期離脱」扱いを基本とする
- プレイヤーへの直接コストは小さく、主に評判低下で効かせる
- 長期離脱後に復帰しない（退団）可能性を持たせる
- 冒険者個性は「能力値」「性格・価値観」「依頼主との相性」を同等重視
- 戦闘は完全自動解決。介入は出発前準備（編成・助言・持ち物）に限定
- 進行は慌てない設計。後のヒアリングで「放置前提にしない」方針へ更新
- 依頼件数は少なすぎない設計（比較検討できる母数を確保）
- AI生成テキストは依頼・冒険者・周辺会話ログ全般に活用
- 人間関係ドラマは分岐イベントより、ログの蓄積で自然発生させる
- ギルド拡張は「大きさの最大化」ではなく「心地よい運営規模への調整」を目的化
- MVPの感情フックは「冒険者がコマではなく人格ある存在だと感じられること」

### 設計原則

- 失敗は排除せず、物語と運用履歴に変換する
- 儲け特化プレイと冒険者重視プレイのどちらも成立させる
- 操作量ではなく判断の質で面白さを作る

## 12. 潜在要件（ヒアリング三次）

### 確定

- 依頼件数は固定日次ではなく、状況に応じて流動的に発生/撤回する
- 依頼更新は定時更新ではなくイベント駆動（持ち込み、取り下げ、営業獲得）
- 冒険者の離脱期間・退団は固定値でなく事由に応じて変動
- 評判は単一値でなく、依頼主側/冒険者側を分けた多軸評価にする
- 評価軸は「適切性」「倫理性」「収益性」など複数指標で管理する
- 失敗ペナルティは固定でなく、失敗理由と文脈によって変動
- 依頼情報は依頼主の自己申告を基本とし、真実とのズレを許容する
- 冒険者能力は数値を直接見せず、経歴・資格・評判・自己評価で判断させる
- 日報UIは「要約3行 + 詳細ログ展開」を基本形にする
- 放置前提の進行はMVPから外す
- AI生成テキストは依頼主/冒険者/日報ログ全体で活用する

### 設計原則（更新）

- プレイヤーが扱うのは「真実」ではなく「観測された情報」
- 管理量の多さではなく、解釈と判断の面白さを主軸にする
- 正解固定の最適化ではなく、価値観ごとの運営スタイルを許容する

## 13. 潜在要件（ヒアリング四次）

### 確定

- `次の日` 押下時の標準順序は「日報確認 -> 日付更新 -> 帰還処理（あれば） -> 新規依頼流入」
- 同時進行案件数の上限は設けない
- 依頼の放置結果は案件ごとの終了条件に依存する
- 掲示板に出す/出さないはプレイヤー裁量で決める
- 依頼主への追加ヒアリングを実装する
- 情報を一部伏せて冒険者へ提案する運用を許可する
- 冒険者からの応募導線を実装する
- 受諾可否はプレイヤーではなく冒険者側の判断で決まる
- 面談ログは要約保存を基本にする
- パーティ編成は冒険者側が最終決定し、ギルドは提案と助言を行う
- 出発後の介入は完全不可
- 結果分類は4段階（成功 / 部分成功 / 失敗 / 惨敗）
- 長期離脱者への見舞い等のフォロー行動を入れる方向
- 評判回復は地道な積み上げを基本とし、一気に回復する手段は置かない
- 評判は数値表示せず、会話・来訪依頼・応募傾向で間接的に表現する
- AI文体は「読みやすさ優先、軽すぎない」トーンを採用する
- MVP必須3画面は「日中運用ハブ / 個別面談 / 帰還日報」とする
- 面談で提示できる依頼件数に実質上限は設けない
- 面談は冒険者側の「もういい」で打ち切り終了する

## 14. 潜在要件（ヒアリング五次）

### 確定

- `次の日` は未読日報がある場合、先に日報要約の確認を必須とする
- 追加ヒアリングは明示コストなしで実行可能
- 追加ヒアリングを重ねすぎると、依頼主の心証悪化や回答品質低下が起きる
- 情報を伏せた提案は、露見かつ失敗に因果がある場合に悪印象を強く発生させる
- 受諾可否は最終的に冒険者が判断し、結果は受諾/保留/辞退で返る
- 出発する/しないの最終決定は冒険者側にある
- 結果4分類の判定は「目的達成度 x 被害度」の2軸で行う
- 長期離脱者への見舞いは、主に冒険者側の心象へ反映する
- 依頼終了理由の代表パターンは「期限切れ / 他ギルド達成 / 依頼主都合」
- AI失敗時フォールバックはMVP対象外（AI前提）
- 面談の長さはギルド側で固定せず、冒険者側の打ち切りで終わる
- 冒険者応募は日次で発生する

### 判定モデル（v0.1）

#### 評判の内部モデル

- 評判は内部で複数軸の隠し値を持つ（表示は非数値）
- 各軸のレンジは `0-100`、中心値は `50`
- 依頼主側軸: `適切性 / 倫理性 / 収益性`
- 冒険者側軸: `安全配慮 / 公正配分 / 成長機会`

#### 冒険者の受諾/保留/辞退判定

- 各冒険者は評価要素ごとの重みを持つ（キャラごとに異なる）
- 評価要素（MVP）:
- `報酬魅力度`
- `申告危険度`
- `依頼主への信頼`
- `依頼内容の明確さ`
- `本人の疲労/気分`
- `ギルドへの信頼`
- `倫理観との一致`
- `過去実績との適合`
- 説得行動は上記総合値に補正を加える（補正量もキャラ依存）
- 判定結果は `受諾 / 保留 / 辞退` の3値

#### 「もういい」打ち切り判定

- 面談中に `面談負荷` を内部加算する
- 負荷要因: 質問回数、同系質問の反復、情報の曖昧さ
- 各冒険者の `許容閾値` を超えると「もういい」で打ち切る
- 追加ヒアリングの心証悪化も同じ負荷モデルで発生する

#### 情報伏せ露見と悪印象

- 露見判定は帰還報告や第三者情報で申告との差分が確認された時に発生
- 悪印象の強化条件: `露見` かつ `失敗に因果あり`
- ただし冒険者の性格・事前同意の有無で悪印象の強さは変動する

#### 結果4分類（固定マトリクス）

- 判定軸は `目的達成度(0-100)` と `被害度(0-100)` を使用
- `成功`: 達成度 >= 80 かつ 被害度 < 40
- `部分成功`: 達成度 >= 50 かつ 被害度 < 70（成功/惨敗条件に該当しない）
- `失敗`: 達成度 < 50 かつ 被害度 < 70
- `惨敗`: 被害度 >= 70 または（達成度 < 30 かつ 被害度 >= 50）

#### 依頼主キャラクターと申告誤差

- 依頼主にも性格タイプを持たせ、申告誤差傾向を変える
- 例:
- `誠実型`: 誤差が小さい
- `誇張型`: 報酬/成果を盛る
- `過小申告型`: 危険を小さく見せる
- 依頼終了理由（期限切れ/他ギルド達成/依頼主都合）の発生傾向も依頼主タイプで変化させる

## 15. 要件優先度（Must / Should / Could）

### Must

- 依頼と冒険者のマッチングが主軸のゲームループ
- 戦闘の完全自動解決
- 依頼のイベント駆動更新（持ち込み/撤回/営業獲得）
- `次の日` 進行順序（標準）を固定する
- 未読日報の要約確認ゲート
- 冒険者の個体性（能力/性格/相性）の可視化
- 冒険者応募と冒険者判断による受諾可否
- 出発可否の最終決定は冒険者側
- 面談件数の上限なし + 冒険者打ち切り終了
- 失敗時の評判低下と冒険者長期離脱
- 出発後の介入不可
- 結果4分類（成功/部分成功/失敗/惨敗）と2軸判定（達成度/被害度）
- 冒険者の個人ログ蓄積（成否・負傷・関係変化）
- 作業感を抑えるための依頼一覧フィルタ/ソート
- 不完全情報UI（自己申告情報、推定、信頼度）
- 評判の二者分離（依頼主側/冒険者側）と多軸評価
- 評判の非数値表現（会話/依頼/応募への反映）+ 内部隠し値運用

### Should

- 依頼主との継続関係（信頼で案件傾向が変化）
- 依頼主への追加ヒアリング
- 連続ヒアリング時の心証悪化（「もういいでしょ」反応）
- 出発前準備の介入要素（装備助言・編成シナジー）
- 情報を伏せた提案運用
- 情報伏せ提案が失敗と結び付いた場合の悪印象強化
- 面談での説得行動（おだてる/応援する/たきつける）
- 退団/復帰揺らぎで喪失体験を発生させる
- 長期離脱者への見舞い/フォロー行動
- 「要約3行 + 詳細展開」の日報UI
- 冒険者との個別面談UI（質問対応、希望確認）

### Could

- 蘇生システム（高コスト儀式や条件付き復帰）
- 都市や派閥ごとの差分ルール
- 任意の軽いオンライン要素（共有ログ/ランキング）

## 16. MVP提案（上記要件ベース）

### MVPゴール

- 「依頼情報が不完全でも、冒険者を見立てて割り当てる判断」が楽しいことを検証する
- 冒険者をコマでなく人格として認識できることを検証する

### MVPで入れる画面（必須3）

- 必須1: 日中運用ハブ画面（依頼ボード + 冒険者一覧 + 進行中案件）
- 必須2: 冒険者個別面談画面（質問対応、提案、助言、受諾判断）
- 必須3: 帰還日報画面（要約3行 + 詳細ログ）

### MVPデータ最小セット

- 依頼: 種別、申告難易度、実難易度、期限、報酬、依頼主信頼度、取り下げ確率
- 冒険者: 経歴タグ、資格タグ、性格タグ、疲労、自己評価、実能力プロファイル
- 評判: 依頼主側（適切性/倫理性/収益性）、冒険者側（安全配慮/公正配分/成長機会）

### MVP進行フロー

1. 日中運用ハブで依頼受理/掲示調整を行う
2. 冒険者との個別面談で質問対応・提案・助言を行う
3. 冒険者判断で受諾/保留/辞退が決まり、出発可否も冒険者が最終決定する
4. `次の日` を押す
5. 未読日報がある場合は先に要約確認（必須）
6. 日付更新 -> 帰還処理 -> 新規依頼流入
7. 評判と個人ログを更新して次の判断へ進む

### MVPで検証するKPI（定性中心）

- 1セッション内で「迷ったが納得して決めた」と感じる判断回数
- 冒険者個人に対する記憶・愛着の自己申告
- 日報詳細ログの閲覧率（要約だけで終わらないか）

## 17. 画面草案（1つ目: 日中運用の3カラム）

### レイアウト

- 左カラム: 依頼ボード（新着/保留/撤回、公開範囲、信頼度メモ）
- 中央カラム: 冒険者一覧（待機中/出発中/離脱中、個別面談の入口）
- 右カラム: 進行中案件と帰還予定（残り日数、要注意フラグ）

### 基本操作

- 依頼をクリック: 詳細確認 -> 掲示方針を変更（全体/限定/保留）
- 冒険者をクリック: 個別面談を開く -> 希望/質問確認 -> 提案/割当
- 進行中案件をクリック: 現在状況と帰還見込みを確認
- 画面右上 `次の日`: 日付を1日進め、帰還報告を発生
- 冒険者の応募通知: 応募理由の要約を確認し、面談へ移動

### 作業感を抑えるための表示ルール

- 常に「今判断が必要なもの」を上位表示（新着依頼、帰還報告、未対応質問）
- 件数が多いときはタブで圧縮（依頼: 新着/進行/保留、冒険者: 待機/出発中/離脱中）
- 日報はポップアップで要約3行のみ先表示し、詳細は任意展開

## 18. 画面草案（2つ目: 冒険者の個別面談）

### 面談画面で表示する情報

- 冒険者プロフィール: 経歴、資格、最近の実績、現在の気分/疲労
- 冒険者の自己申告: 「この依頼は得意/不安」などの主観コメント
- ギルド側メモ: 過去配属との相性、注意点、依頼主との関係
- 候補依頼: この冒険者に提案可能な依頼一覧（公開範囲内）

### 面談中に選べる行動

- 質問に答える: 依頼内容の追加説明、報酬/危険説明
- 提案する: 特定依頼への応募を打診する
- 見送る: 今回は配属せず待機を選ぶ
- 助言する: 装備方針、同行推奨、撤退優先などの出発前方針
- 説得する: おだてる / 応援する / たきつける などの働きかけ
- 打ち切られる: 冒険者が「もういい」を選ぶと面談終了

### 面談結果として保存する内容

- 冒険者の意思: 受諾/保留/辞退
- 信頼変化メモ: 納得感が高い説明だったかどうか
- 次回面談フラグ: 再提案したい依頼、要注意条件
- 発言要約: 面談内容の要点だけを短文保存
- 打ち切り理由要約: 冒険者側が面談を終えた理由（短文）

### UI方針（作業感対策）

- 1画面1人に集中し、同時に大量情報を見せない
- 重要情報（受諾可否に影響する要素）を上段固定表示
- 面談ログは短文で蓄積し、台帳から後で参照可能にする

## 19. 画面草案（3つ目: 帰還時の日報テンプレ）

### 1. 要約3行の固定フォーマット

- 1行目（結論）: `依頼名 / 成否 / 主要成果`
- 2行目（被害）: `パーティ状態（無事・負傷・長期離脱）/ 消耗状況`
- 3行目（余波）: `依頼主反応 / 評判変化の要点 / 次回メモ`

### 2. 詳細ログに必ず載せる項目

- 依頼情報: 依頼名、依頼主、受注日、帰還日
- 配属情報: 参加冒険者、役割、出発前助言
- 結果情報: 成功/部分成功/失敗/惨敗、獲得物、損耗、想定外イベント
- 反応情報: 依頼主コメント、冒険者コメント
- 更新情報: 評判変化、個人履歴への追記内容

### 3. 要注意アラートの表示条件

- 冒険者が長期離脱または退団した
- 依頼主側または冒険者側の評判が大きく低下した
- 同種依頼で失敗が連続した
- 申告情報と実態の乖離が大きい依頼主が判明した

### 4. 評判・個人履歴への反映ルール（文言テンプレ）

- 評判（依頼主側）: `適切性 +/-, 倫理性 +/-, 収益性 +/-`
- 評判（冒険者側）: `安全配慮 +/-, 公正配分 +/-, 成長機会 +/-`
- 個人履歴（冒険者）: `誰の依頼で / 何を担当し / どう感じたか` を短文で追記
- メモ方針: 失敗時も責任追及のみで終わらせず、次回判断に使える学習文を残す

## 20. MVP実行提案（v1）

### 目的

- 「不完全情報での采配」が1プレイで体験できる状態を最短で作る
- 冒険者を人格ある存在として認識できる最小要素を成立させる

### MVPスコープ（やること）

- 日中運用ハブで、依頼受理/掲示調整/応募確認ができる
- 冒険者個別面談で、質問対応/提案/助言/説得ができる
- 冒険者判断で受諾/保留/辞退が決まる
- `次の日` 押下で時間が進み、帰還案件は日報で報告される
- 未読日報要約の確認前は `次の日` を進められない
- 結果は4分類（成功/部分成功/失敗/惨敗）で表示される
- 評判は非数値で、会話文と依頼/応募傾向に反映される

### MVPスコープ（やらないこと）

- 戦闘の直接操作
- 派閥/都市ごとの特殊ルール
- オンライン要素
- 蘇生など重い復帰システム

### 開発順（3ステップ）

1. ループ成立
- 日中運用ハブ、`次の日` 進行、帰還日報の最小導線を実装

2. 判断体験の成立
- 個別面談、追加ヒアリング、情報伏せ提案、冒険者受諾判断を実装

3. 物語性の成立
- 要約ログ蓄積、評判の非数値反映、応募理由/依頼主反応テキストを実装

### MVP観察ポイント（判定ゲートなし）

- 1サイクル（受理 -> 面談 -> 出発 -> `次の日` -> 帰還報告）を連続3回プレイして破綻しない
- 1回以上「冒険者が受諾しない」ケースが発生し、理由がログで理解できる
- 1回以上「失敗 or 惨敗」が発生し、評判や次回行動に影響が見える
- 1人の冒険者に対して、履歴ログから関係の変化を追える

### 最初のデモシナリオ

- 依頼数: 12件（公開8/限定3/非公開1）
- 冒険者数: 6人（志向が分かれる構成）
- プレイ時間: 15〜20分で1日サイクルを2〜3回回せる
