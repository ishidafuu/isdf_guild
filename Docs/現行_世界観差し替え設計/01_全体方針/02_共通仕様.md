# 世界観差し替え型ギルドゲーム 仕様草案

最終更新: 2026-03-11

## 1. 基本方針

本ゲームは、ギルド運営と依頼解決をコア体験とする。
世界観はコアルールの上に載る差し替えレイヤーとして扱い、ゲームの進行構造そのものは変えない。
各世界観は完全に別世界として扱い、多元世界の別地域として無理に接続しない。

本仕様では以下を目的とする。

- メインループを固定する
- 世界観ごとの差分をモジュール化する
- 人力制作と自動生成の両方に対応する
- 世界観追加時に既存ルールを壊さない

## 2. ゲームのコア構造

以下の要素は全世界観で共通とし、変更しない。

- プレイヤーはギルド所属キャラクターを運用する
- ギルドは依頼を受注し、達成によって資源を得る
- 資源を用いてキャラクターおよびギルドを成長させる
- セッションは依頼単位で進行する
- 成否に応じて報酬・損害・世界変化が発生する
- ギルド内には、依頼へ直接出ない運営スタッフを置ける

補足:
本作は、物語性の高い経営ゲームを目指す。
成長の主役はキャラクター同士の物語と関係変化に置き、ギルド拡張は任意の成長軸として扱う。
プレイヤーが拡張したければ大きくでき、したくなければ現状維持寄りでも遊べる設計を目指す。
個人成長とギルド成長のどちらか一方を強く押し出すよりも、両方がゆるやかに変化していく状態を基本とする。
王道ファンタジーは基準世界ではなく、初期候補群のうちの並列な1世界観として扱う。
ギルド規模は「大きくすれば正解」にはせず、大きくなったなりの利点と重さが両方増える設計を目指す。
街や組織の変化は、数字の上昇そのものよりも、空気、立場、人間関係、受けられる仕事の変化として感じられる形を優先する。
このゲームの主役は、依頼、キャラクター、ギルドのどれか単体ではなく、それらが絡み合って立ち上がる全体の物語である。
最も避けたい失敗は、物語が消えて作業的なレベル上げや効率化だけが残ることである。
ギルド内の運営スタッフは、数値的な補助要員ではなく、提案、人物評、方針への口出しを行う別視点として扱う。

## 3. メインループ

全世界観で共通のゲーム進行は以下とする。

1. 依頼提示
2. 依頼選択
3. 準備
4. 任務遂行
5. 判定解決
6. 結果反映
7. 報酬獲得
8. 成長・拠点更新
9. 次回依頼の解放

補足:
各世界観は、この流れの名称や演出を変更してよいが、段階の機能は変更してはならない。
また、各段階の前後で運営スタッフから短い提案、懸念、人物評が挟まってよい。

## 4. コアルール固定領域

以下は世界観による変更不可領域とする。

- 判定方式
- 難易度の段階構造
- 成功 / 失敗 / 大成功 / 部分成功などの結果区分
- キャラクター成長の基本枠
- ギルド成長の基本枠
- 依頼カテゴリの機能分類
- リソース管理の基礎構造
- 失敗時の処理原則
- セッション進行順

## 4.1 現時点の固定方針

以下は、2026-03-11 時点で暫定的に固定する方針である。

- 判定方式は `2D6` とする
- 判定段階は `大成功 / 成功 / 部分成功 / 失敗` の4段階とする
- 戦闘は原則として「依頼中の1障害」として軽く扱う
- 1セッションの基本単位は軽めの短編依頼1本とする
- 短編依頼1本の想定時間は 15 分から 30 分程度とする
- 依頼1本の標準構成は 3 場面程度とする
- 死亡や引退などの恒久損失は、固定ルールではなくゲーム開始時の設定項目として切り替えられるようにする
- 負傷やストレスは、原則としてセッションをまたいで残るものとする

## 4.2 キャンペーン設定項目

世界観パックとは別に、ゲーム開始時にキャンペーン全体の温度感を決める設定項目を持たせる。

例:

- 恒久損失の有無
  - 死亡あり / 引退のみ / 恒久離脱なし
- 負傷の重さ
  - 軽め / 標準 / 重め
- 依頼失敗の重さ
  - 軽め / 標準 / 重め
- 物語トーン
  - 明るい冒険 / 苦い群像劇 / 混在 / ランダム
- コメディ許容度
  - 低い / 標準 / 高い
- 残酷さ・理不尽さ
  - 低い / 標準 / 高い
- 中心感情
  - 達成感 / 哀愁 / 緊張 / 連帯感 / 成り上がり感 / 世界の謎への接近 / 混在

補足:
これにより、同じコアルールでも「人が簡単には死なない群像劇」と「人が欠けていく苦い運用劇」の両方を扱えるようにする。
また、作品全体として「この遊び方をしてほしい」と固定しすぎず、トーン自体をプレイヤーや作者が選んだり、ランダム生成したりできるようにする。

## 4.3 ギルド規模の考え方

ギルド規模の拡大は、単純な上位互換として扱わない。
規模が大きくなるほど、受けられる仕事や影響力が増える一方で、抱える責任や内部問題も増えるようにする。

規模拡大で増えるものの例:

- 依頼数の増加
- 上位案件や特殊案件への接続
- 派閥や権力との接点
- 抱える人員と設備の幅

規模拡大で増える負担の例:

- 内部の不和や人間関係のもつれ
- 外部からの期待や干渉
- 守るべき評判の重さ
- 人手不足や管理負担の深刻化

補足:
理想は、ユーザーが「自分にとって心地よい規模」を探っていけることにある。

## 4.4 ギルドスタッフの位置づけ

ギルドには、冒険者とは別に、依頼へ直接出ない運営スタッフを置けるようにする。

主な役割:

- 依頼前に人選や危険について提案する
- 依頼後に冒険者への人物評を返す
- 平時にギルド方針への意見を出す
- 休養や交代について口を出す

補足:
スタッフは 0 人でも遊べるが、1 人から 3 人程度いるとギルド内の会話と視点が増えやすい。
主役はあくまで全体の物語であり、スタッフはそれを補助する存在として扱う。

## 5. 共通データ構造

世界観差し替えに対応するため、内部的には機能名で管理する。

### 5.1 能力値

能力値名は世界観ごとに表示変更可だが、内部機能は固定とする。

- `power`
  - 武力、制圧、突破
- `tech`
  - 工作、修理、分析
- `sense`
  - 察知、追跡、探索
- `social`
  - 交渉、説得、取引
- `will`
  - 精神抵抗、集中、継戦

### 5.2 ロール機能

- `frontliner`
  - 前線役。脅威を受け止め、排除し、突破口を作る役割
- `support`
  - 支援役。回復、強化、防護、継戦補助を担う役割
- `scout`
  - 探索役。索敵、追跡、先行確認、異常発見を担う役割
- `engineer`
  - 技術役。修理、解除、分析、特殊装置対応を担う役割
- `negotiator`
  - 渉外役。交渉、説得、聞き込み、利害調整を担う役割

表示名は世界観ごとに変更可能。

## 5.2.1 性格骨格

キャラクターの内面的な芯として、BIG5 相当の性格骨格を共通構造に含める。

- `openness`
  - 開放性。未知への志向、柔軟さ、新奇好み
- `conscientiousness`
  - 誠実性。準備、継続、責任感、几帳面さ
- `extraversion`
  - 外向性。前に出る姿勢、対人接触、活発さ
- `agreeableness`
  - 協調性。共感、譲歩、他者配慮
- `neuroticism`
  - 神経症傾向。不安、過敏さ、動揺しやすさ

補足:
これは能力値とは別系統の内部骨格であり、戦闘力ではなく人物の判断傾向や会話傾向を安定させるために使う。
性格タグや欠点は、この骨格を読みやすい形で表へ出すレイヤーとして扱う。

### 5.3 依頼カテゴリ

以下は共通カテゴリとして保持する。

- `subjugation`
  - 討伐。脅威を排除、無力化、鎮静化する依頼
- `escort`
  - 護衛。人物や物資を安全に守る依頼
- `investigation`
  - 調査。原因、真相、所在、証拠を明らかにする依頼
- `recovery`
  - 回収。物品、情報、人材、遺物などを確保する依頼
- `delivery`
  - 輸送。対象物を期限内に運ぶ依頼
- `negotiation`
  - 交渉。対立相手との合意や取引成立を目指す依頼
- `defense`
  - 防衛。拠点や人物を一定時間守りきる依頼
- `rescue`
  - 救助。危機にある対象を発見し保護する依頼

各世界観は表示文言を差し替えてよい。

### 5.4 基本リソース

- `hp`
  - 体力。負傷や損耗の蓄積を表す
- `supply`
  - 物資。消耗品、補給、任務継続資源を表す
- `money`
  - 資金。金銭や等価資源を表す
- `reputation`
  - 評判。信用、名声、組織評価を表す
- `action` または `fatigue`
  - 行動余力または疲労。1任務中の継続力を表す

表示名や演出名のみ変更可能。

## 6. 世界観パックの責務

世界観パックは以下を定義するモジュールとする。

- 世界の時代感
- ギルドの立場
- 主な脅威
- 勢力一覧
- 拠点の外観
- 依頼カテゴリの表示名
- ロール名の表示名
- 報酬の名称
- 消耗品や装備の名称
- ランダムイベント表
- 敵カテゴリ表
- 導入文と文体
- 固有フレーバー要素

補足:
世界観パックは、ルール処理ではなく表示・解釈・演出を担当する。
ただし、同じコアループを保つ範囲で、依頼の手触りや判断の重みなどのプレイ感に差が出ることは許容する。

## 7. 世界観が変更してよい範囲

以下は世界観差分として自由に変更可能。

- 用語
- テキスト文体
- 依頼文
- 敵の見た目と分類名称
- 勢力名称
- 通貨や報酬の呼称
- 拠点名称
- 特殊イベント表
- 状態異常や損害の演出名
- 特技や装備のフレーバー記述

## 8. 世界観が変更してはいけない範囲

以下を変更するとゲーム互換性が壊れるため禁止とする。

- 判定の根本式
- 成長曲線の基本設計
- 依頼1件あたりの進行構造
- ロール機能の本質
- 難易度と報酬の対応原則
- 失敗時の代償発生構造
- ギルド成長の主要ループ

## 9. 世界観パックの最小構成

1つの世界観パックは、最低限以下を含める。

- パック名
- 一文説明
- 世界説明 3行
- ギルド説明 3行
- 勢力 3つ以上
- 依頼カテゴリ言い換え一覧
- ロール名言い換え一覧
- 敵カテゴリ 8件以上
- イベント表 6件以上
- 報酬名称
- 拠点名称
- 導入シナリオ1本分の種

補足:
初期ワールドの本数は、先に固定しない。
まずは代表的なパックをいくつか用意して実際に動かし、別の世界観も十分に回せると判断できた段階で追加していく。

推奨の初期試験パック:

- 王道ファンタジー
- 退廃サイバーパンク
- 和風怪異

## 9.1 世界観差の許容範囲

世界観差は、見た目や語彙だけに限定しない。
同じコアループを維持する限り、以下の差分は許容する。

- 依頼で重視される障害の種類
- 報酬の価値観
- 損害や余波の重み
- 交渉と戦闘の比重
- 情報秘匿や因習の濃さ

補足:
ただし、差分が大きすぎて別ゲーム化しないよう、判定方式と依頼処理構造は固定する。

## 10. 世界観生成テンプレート

自動生成または半自動生成用に、以下の入力項目を定義する。

- `era`
  - 中世 / 近代 / 現代 / 未来 / 終末後 など
- `scale`
  - 都市 / 地方 / 国家 / 宇宙圏 など
- `guild_role`
  - 請負組織 / 互助会 / 調査機関 / 交易団 など
- `threat`
  - 魔物 / 怪異 / 企業 / 機械 / 災害 / 海賊 など
- `mission_tone`
  - 戦闘寄り / 探索寄り / 交渉寄り / 輸送寄り
- `reward_value`
  - 金 / 物資 / 信頼 / 名声 / 権利 / 情報
- `base_type`
  - 酒場 / アジト / 社務所 / 船 / 宇宙港 / 学園部室
- `mood_tags`
  - 王道 / 怪奇 / 退廃 / 青春 / 哀愁 / 開拓
- `taboo`
  - その世界で出したくない要素
- `unique_hook`
  - その世界固有の1要素

## 11. 世界観生成ルール

生成時は以下の原則に従う。

- コアループに合わない設定は採用しない
- 依頼カテゴリ8種に自然に変換できること
- ギルドという組織形態が成立していること
- 報酬概念が成立していること
- 拠点成長の余地があること
- 主要脅威が依頼化可能であること

## 12. 品質チェック条件

新規世界観パックは以下を満たす場合のみ採用する。

- 依頼カテゴリ8種のうち6種以上に自然な名称を与えられる
- ロール機能5種すべてに対応する役割名を与えられる
- 報酬、敵、勢力、拠点がそれぞれ明確に定義されている
- 任務失敗時の不利益を世界観上自然に説明できる
- 既存世界観と十分に雰囲気差がある
- 1セッション分の導入を即座に作れる

## 13. 実装上の推奨設計

データ管理は以下の分離を推奨する。

- `core`
  - 判定、進行、成長、依頼処理
- `world_pack`
  - 用語、勢力、敵表、依頼文、イベント表
- `scenario`
  - 個別シナリオ、地域差分、NPC配置

補足:
将来的な実装では、世界観辞書、依頼生成表、特技辞書などは JSON やデータテーブルに落とす前提で整理する。
現在の Markdown 文書は、その前段階の人間可読な正本として扱う。

## 13.1 判定処理の現時点方針

現時点の判定処理は以下の方針で整理する。

- 基本式:
  - `2D6 + 能力値 + 補正 >= 目標値`
- 判定結果:
  - 目標値より 2 以上高い: 大成功
  - 目標値以上: 成功
  - 目標値より 1 低い: 部分成功
  - 目標値より 2 以上低い: 失敗

補足:
部分成功は「目的を一部達成するが、代償や不足が残る」結果として扱う。

### 13.1.1 目標値の標準レンジ

軽量運用を優先し、目標値は広げすぎない。
まずは以下の 5 段階を標準とする。

- 6
  - 易しい。得意分野ならまず通せる
- 7
  - 標準。平均的な業務難度
- 8
  - やや難しい。得意分野でないと不安が出る
- 9
  - 難しい。準備、特技、相性補正が欲しい
- 10
  - 厳しい。重要局面や不利条件込みの山場

補足:
11 以上は、通常依頼ではなく重大案件や悪条件の重ねがけとして扱う。
このゲームでは判定自体を主役にしないため、目標値は厳密なチューニング対象というより「ざっくり結果を振り分ける段階」として扱う。

### 13.1.2 補正の考え方

補正は細かく積み上げすぎず、基本は小さな整数で扱う。

- `+1`
  - ちょっと有利。適性がある、準備が刺さる、関係が良い
- `+2`
  - 明確に有利。強い特技、十分な準備、状況が噛み合っている
- `-1`
  - ちょっと不利。疲労、軽傷、時間不足、相性の悪さ
- `-2`
  - 明確に不利。欠点の発動、重傷、情報不足、強い妨害

補足:
原則として 1 回の判定で補正は `-2` から `+2` 程度に収める。
大きな差は数値で盛るより、判定回数や必要条件で表現する。
これは十分な標準幅として扱い、細かい補正の積み上げは避ける。

### 13.1.3 部分成功の共通代償

部分成功は、依頼が止まらない代わりに何かを失う結果として扱う。
代償は以下のいずれか 1 つを基本とする。

- 時間を失う
  - 追加場面や期限悪化が発生する
- 物資を失う
  - `supply` や任務道具を消耗する
- 状態が悪化する
  - 負傷やストレスが 1 段階進む
- 関係が揺らぐ
  - 同行者、依頼人、勢力との関係が悪化する
- 情報が欠ける
  - 真相の一部を取り逃がす、後続調査が必要になる
- 成果が不完全になる
  - 救助人数、回収量、防衛達成度などが落ちる

補足:
短編依頼では「前に進むが、何かが残る」を優先し、足止めより余波を作る方向で使う。
ここで挙げた代償カテゴリを、部分成功の基本セットとして採用する。

### 13.1.4 大成功の共通利益

大成功は、単なる成功に加えて次のどれか 1 つの追加利益を与える。

- 報酬増加
- 関係改善
- 追加情報の獲得
- 状態悪化の打ち消し
- 次の判定への有利
- 隠し問題の先取り解決

補足:
大成功は爽快感だけでなく、後続の物語を滑らかにする報酬として機能させる。
ここで挙げた利益カテゴリを、大成功の基本セットとして採用する。

### 13.1.5 標準依頼での判定回数

軽めの短編依頼 1 本では、判定回数を多くしすぎない。
標準的には以下を目安とする。

- 主要判定 2 回から 4 回
- 補助判定 0 回から 2 回
- 山場でも合計 5 回前後まで

補足:
戦闘はこの中の 1 障害として処理し、専用の長いサブゲームにはしない。

### 13.1.6 判定前に参照する要素

判定前に見る要素は、基本的に以下に絞る。

- 能力値
- 特技
- ロール適性
- 欠点発動の有無
- 関係性
- 状態値
- 世界観固有の事情

補足:
キャラクターの人格と現在状態が毎回少しずつ結果へ混ざる構造を基本とする。

### 13.1.7 特殊出目の扱い

現時点では、ゾロ目や特定出目そのものに特別ルールは設けない。

補足:
本作において判定は、戦闘や任務の細部を演出する主役ではなく、依頼進行を裏で前に進める軽量処理として扱う。
そのため、出目固有イベントよりも、成功段階と余波の整理を優先する。

## 13.2 負傷・ストレス・休養の現時点方針

負傷とストレスは、短編依頼をつなぐ継続要素として扱う。
どちらも数値肥大より段階管理を優先する。

### 13.2.1 状態段階

まずは 4 段階を標準とする。

- 0: 問題なし
- 1: 軽度
- 2: 中度
- 3: 重度

負傷とストレスは別管理とするが、構造は共通化する。

補足:
段階数そのものは絶対条件ではなく、全体の扱いやすさを優先して決める。
現時点では `0-3` の 4 段階を標準案とする。

### 13.2.2 状態悪化の主な契機

- 失敗や部分成功の代償
- 無理な配置
- 欠点発動下での強行
- 休養不足
- 重大事件や喪失

### 13.2.3 状態が判定へ与える影響

- 軽度
  - 特定場面で `-1` がつくことがある
- 中度
  - 関連判定で `-1` が常態化しやすい
  - 欠点や感情の表面化条件が増える
- 重度
  - `-2` 相当の不利が出る
  - 一部依頼への参加を避けた方がよい
  - 恒久損失設定によっては離脱判定候補になる

### 13.2.4 回復手段

状態回復は主に以下で行う。

- 休養
  - 依頼に出さず時間を与える
- 人間関係イベント
  - 会話、和解、見送りなどでストレスを和らげる
- 設備利用
  - 医務室、相談室、工房などの拠点機能を使う
- 特技支援
  - 応急処置、精神支援などの特技で軽減する
- 報酬や外部支援
  - 医療資源、祈祷、技術補助など世界観相応の手段を使う

補足:
完全自動回復を標準にせず、「誰を休ませるか」が人材運用になるようにする。
自然回復は速すぎない方針とし、毎回機械的に 1 段階ずつ戻る前提にはしない。
長期離脱や復帰待ちも、物語の一部として許容する。

### 13.2.5 休養の位置づけ

休養は空白ターンではなく、物語の一部として扱う。

- 休ませた結果、関係性が変わる
- 休養中に個別イベントが起きる
- 出番が減る代わりに状態が戻る
- 無理に出すと成果は出ても後が重くなる

補足:
休養中イベントは「回復のために選ぶ専用行動」ではなく、ギルドにいる時間の中で自然に起こる会話や出来事として扱う。
その結果として、たまたまストレスが和らぐ形を推奨する。

### 13.2.6 重度状態の扱い

重度状態のキャラは、強い不利に留めるだけでなく、状況によっては依頼参加不可として扱ってよい。

- 重傷で現場に出せない
- 強いストレスで任務適性を失っている
- 本人は出たがっているが、ギルド側が止めるべき状態

補足:
「まだ戻れない」「早く復帰したい」といった停滞も、人物描写として許容する。

## 13.3 関係値運用の現時点方針

関係値は数値を細かく積むより、少数段階とタグで扱う。

### 13.3.1 基本構造

各キャラは以下の関係を持てる。

- ギルドとの関係
- 他キャラとの個別関係
- 主要 NPC との関係
- 勢力との関係

表現は次の 2 層を推奨する。

- 関係段階
  - `-2` 不和
  - `-1` 緊張
  - `0` 中立
  - `+1` 良好
  - `+2` 深い信頼
- 関係タグ
  - 例: `借りがある` `対抗心` `保護対象視` `気まずい` `恩人`

補足:
数値だけでは物語が痩せるため、タグ併用を前提とする。
関係の種類そのものが重要であるため、同じ段階でもタグによって意味が変わる前提で扱う。

### 13.3.2 関係が動く契機

- 同行依頼の成功
- 失敗の責任
- 救助や支援
- 約束の履行や破り
- 欠点発動による衝突
- 休養中の会話イベント
- 派閥や依頼人を巡る利害対立

### 13.3.3 関係がゲームへ与える影響

- 良好関係
  - 補助判定で `+1`
  - 部分成功時の代償をかばえる
  - 追加情報や追加選択肢が開く
- 悪化関係
  - 連携不全や判断齟齬が起きやすい
  - 欠点発動条件が増える
  - 一緒に出すと別の火種や会話イベントが混ざる

補足:
悪い関係は、常に単純な不利補正として処理するのではなく、不安定さや物語上の変化要因として扱う。
危機によって改善することもあれば、余計に悪化することもある。

### 13.3.4 関係の運用原則

- すべてを数値で処理しない
- 「今この2人を同じ案件に出すとどうなるか」が見えることを重視する
- 改善も悪化も、依頼外イベントで起こせるようにする
- 関係は報酬や成長の主役として扱う
- 関係の悪さは、必ずしも悪影響だけでなく、改善や反転の契機にもなりうる

## 13.4 軽量ギルド運営の接続方針

判定と人物運用をつなぐため、ギルド状態は細かな会計ではなく少数の状態で持つ。

- 評判
  - 依頼の幅と世間の見方
- 人手余力
  - 誰をどれだけ無理なく回せるか
- 設備充実度
  - 回復、調査、工作の下支え
- 外部関係
  - 勢力や依頼人との導線

補足:
ギルドが大きくなると、依頼数や政治性だけでなく、管理負荷や火種も増える。
理想は「大きくするかどうか」自体がプレイヤーの物語判断になることである。

### 13.4.1 拠点拡張の見せ方

拠点拡張は、数値レベルを積む方式と、設備や部屋が増える方式のどちらでも成立する。
ただし現行方針では、後者の方が相性がよい。

#### A. 数値レベル型

- 例:
  - 拠点レベル 1 / 2 / 3
  - 収容力、設備力、信用力などが段階上昇する

利点:

- 管理しやすい
- AI やロジックで参照しやすい
- 進行度が一目でわかる

弱点:

- 「数字を育てている感」が前に出やすい
- 今回の物語重視方針とは少しズレやすい

#### B. 設備一覧型

- 例:
  - 医務室がある
  - 記録庫がある
  - 相談室がある
  - 工房がある
  - 客間がある

利点:

- 拠点の変化を物語として感じやすい
- キャラ運用や日報描写に直結しやすい
- 世界観ごとの差し替えとも相性がよい

弱点:

- 実装時に設備ごとの効果整理が必要
- 一覧が増えすぎると見通しが落ちる

#### C. 状態 + 設備の折衷型

現行方針としては、以下の折衷が最も有力である。

- 内部的には少数の状態で持つ
  - 評判
  - 人手余力
  - 設備充実度
  - 外部関係
- 表示上は設備一覧や部屋追加として見せる

補足:
この形なら、内部処理は軽く、見た目は物語的に変化させやすい。

#### 現時点の方針

- プレイヤーへの見せ方は `設備一覧型` を優先する
- 内部ロジックは少数状態値との併用を許容する
- つまり、実装上は折衷型でもよいが、プレイヤー体験としては `拠点レベル` より `設備や部屋が増えていく感覚` を前に出す
- 各設備はレベル表記よりも、短い説明文と機能の組み合わせで表現する
- 設備内容は世界観ごとに柔軟に差し替えてよい
- 完全に共通設備へ寄せすぎず、その世界観で自然な施設や部屋を優先してよい
- 拠点拡張は「数字が上がる」よりも、以下の変化を感じられることを重視する
  - 受けられる依頼が増える
  - キャラ同士の会話や休養イベントが増える
  - 外部とのつながりが増える
  - ギルドの雰囲気が変わる
- データ上の影響が小さい設備でも、物語や雰囲気を深めるなら導入してよい

補足:
実際の試運転を通して、世界観をまたいで共通的に有用な設備が見えてきた場合は、その時点で共通設備候補を再整理する。

## 13.5 依頼生成運用方針

依頼カテゴリは、現時点では 8 種で維持する。
採取や潜入の独立カテゴリ化は、実際の試運転後に必要性を見て再判断する。

また、依頼生成は最終的に完全ランダムでも成立するよう設計する。
ただし、生成結果の品質確認や手直しを妨げるものではない。

依頼内容の方針:

- 毎回必ず隠し事や追加事情を入れる必要はない
- 軽い日常依頼も許容する
- 重い依頼と日常依頼の両方が混ざることで、ギルド運営らしい流れを作る
- 失敗時の余波の重さは、依頼内容と関係者に応じて変える
- 依頼人や勢力との関係により、案件内容や条件が変質する構造は強めに持たせてよい

## 13.6 AI 生成の基本方針

本作は、ゲーム開始後の進行については AI 生成を主軸に置く。

AI に主に任せる対象:

- 依頼文
- NPC
- 日報
- キャラ背景
- 世界観の個別事件や細部

人間が主に決める対象:

- ゲーム開始時の世界観の大枠
- 遊びたくない世界観を除外する条件
- 開始時のトーンやキャンペーン設定

補足:
ゲーム開始後は、可能な限り AI に進行を委ねる前提とする。
作者やプレイヤーは入口の方向性を決め、個別の展開生成は AI が担う。

### 13.6.1 AI とロジックの責務分離

旧設計から継承する有効な方針として、AI とロジックの責務は分ける。

- AI が担当するもの
  - 人物文
  - 依頼文
  - 会話応答
  - 日報文
  - 街やギルドの空気を表す短文
- ロジックが担当するもの
  - 判定結果
  - 関係値や状態値の更新
  - 報酬、損害、タグ更新
  - 進行順序

補足:
ゲーム開始後は AI 主導で進める方針を維持しつつ、状態更新の正本まで AI へ委ねない。
これにより、文体の自由さと進行の安定性を両立させる。

### 13.6.2 主な AI 生成イベント

現行設計でも、以下の生成イベントはそのまま再利用しやすい。

- キャラクター初期生成
- 依頼人や主要 NPC の初期生成
- 依頼文生成
- 面談や会話応答生成
- 出発判断コメント生成
- 帰還日報生成
- 評判や噂の反映テキスト生成

補足:
人物、依頼、日報の 3 系統を優先し、必要に応じて街の噂や周辺反応を足す。

### 13.6.3 出力契約の原則

AI 出力は、できるだけ機械処理しやすい形に寄せる。
最低限、以下のような構造を推奨する。

- `text`
  - 表示本文
- `intent_tags`
  - 応答意図や感情のタグ
- `reason_summary`
  - 短い理由要約

補足:
人物年表や日報要約に再利用しやすくするため、本文だけでなく短い要約項目を持たせるのがよい。

### 13.6.3.1 キャラクター生成時の推奨出力

キャラクター生成では、少なくとも以下の 3 層を返すことを推奨する。

- `public_digest`
  - 一覧や面談で見せる短文
- `private_dossier`
  - 内部人物資料
- `volatile_hook`
  - 最近の悩みや変動要素

補足:
`private_dossier` には、生い立ち、家族関係、価値観、対人傾向、仕事観、過去の転機などを含めるとよい。
これにより、能力値だけでは出ない人物差を、依頼反応や会話へ反映しやすくなる。

### 13.6.3.2 AI 出力契約の固定度

AI 出力契約は、どこまで厳密に固定するかで運用負荷が変わる。
現時点では次の 3 段階で考えられる。

#### A. ゆるい固定

- 必須は `text` のみ
- あとは自由文中心

利点:

- 早く試せる
- 文体の自由度が高い

弱点:

- ログ再利用や UI 連携が不安定になる
- AI 出力の揺れを吸収しにくい

#### B. 中程度固定

- 必須:
  - `text`
  - `intent_tags`
  - `reason_summary`
- 任意:
  - `journal_headline`
    - 日報の見出し
  - `emotion_phrase`
    - その場の感情を短く示す語句
  - `lesson_tags`
    - 今回の出来事から残った学びや傾向タグ

利点:

- 表示と再利用のバランスがよい
- 人物年表や日報、面談要約へ流し込みやすい
- 今の仕様密度に合う

弱点:

- ある程度の JSON バリデーションが必要

#### C. 厳密固定

- イベントごとに完全な JSON スキーマを固定
- フィールド不足や揺れを厳しく弾く

利点:

- 実装しやすい
- 後工程が安定する

弱点:

- 仕様変更コストが上がる
- 試行錯誤の早さが落ちる

#### 現時点の方針

- 今は B の中程度固定を採用する
- つまり、本文に加えて「意図タグ」と「短い理由要約」は必須にする
- 補助項目は初期必須にはせず、必要になった時点で追加する
- 出力契約を先に固める対象は、現時点では `日報` を優先する

補足:
日報は継続プレイ時の接着剤になりやすく、`text`、`intent_tags`、`reason_summary` の組み合わせを試す対象として最も自然である。
また、初期段階では「何が起こったか」という事実ログが取れれば十分であり、見出しや感情語句はプレイヤー表示の必須要件にしない。
内部的に必要な反応や学びは、ログや人物状態へ反映されていればよい。

### 13.6.4 品質ガード

旧設計から継承できる品質方針として、以下を採用する。

- AI 出力は表示文と理由要約までに留める
- 数値更新は AI 出力に依存しない
- 長文より、要約と詳細の二段構成を優先する
- キャッシュや再試行を前提とする
- 失敗時も完全な固定テンプレに落としすぎない

補足:
本作の価値は物語の手触りにあるため、AI 不調時も味気ない固定文へ即座に置換しない方がよい。

### 13.6.5 日報の基本形

帰還日報は、旧設計の「要約 3 行 + 詳細展開」を継承する価値が高い。

- 1 行目
  - 依頼名 / 成否 / 主要成果
- 2 行目
  - パーティ状態 / 消耗 / 負傷
- 3 行目
  - 依頼人反応 / 評判変化 / 次回メモ

補足:
軽い依頼を積み重ねる設計と相性が良く、プレイヤーが毎回全詳細を読まなくても流れを追いやすい。

### 13.6.5.1 ギルド主メモの基本形

日報や面談の後には、事実ログとは別に `guildmaster_note` を残せるようにする。

現時点の方針:

- `guildmaster_note` は AI が 3 件の仮メモを提示する
- プレイヤーは最も近いものを 1 件選ぶ
- 必要なら短文補足を追加する
- その結果を人物メモとして保存する

補足:
これは AI が勝手に人物評価を確定するためのものではなく、
プレイヤーがその人物をどう見たかを軽く残すための仕組みである。
事実ログは AI が持ち、主観メモはプレイヤーが選ぶ構造に分ける。

### 13.6.6 進行ゲートとしての日報確認

未読の日報要約がある場合、次の進行前に一度確認させる構造は現行設計でも有効である。

補足:
これはテンポを壊すためではなく、人物と依頼の余波を見落とさないための最小ゲートとして使う。

## 13.7 想定プレイ形態

本作は、現時点ではソロプレイ前提で設計する。

- プレイヤーは 1 人
- 自分のギルドと、その周囲の人物や依頼の物語を楽しむ
- 複数人TRPG運用は、少なくとも初期設計の主軸には置かない

補足:
プレイヤーが「自分が作った物語だった」と感じられることを重視する。

## 14. 設計思想の要約

本ゲームは「世界観固定の作品」ではなく、
「ギルド運営ゲームのコアシステムに複数の世界観を載せるフレームワーク」として設計する。

そのため、開発上の優先順位は以下とする。

1. メインループを壊れない形で定義する
2. 共通機能名を先に固定する
3. 世界観差分は表示と演出に寄せる
4. 必要最小限の世界観パック仕様を定める
5. 追加や生成が容易な構造を保つ
