# フェーズ1 型と正準データ構造の固定

最終更新: 2026-03-10

このフェーズでは、仕様書上の JSON 叩き台を TypeScript の正準型へ落とし込む。
目的は、以後のフェーズでデータ構造の解釈がぶれない土台を作ることにある。

## 1. このフェーズの目的

- JSON 叩き台をコード上の型へ固定する
- 命名規則と ID 規則をコード側でも統一する
- 後続フェーズの入出力を曖昧にしない

## 5. 作業項目

### 5.1 共通 ID と命名の切り出し

- `character_id`
- `mission_id`
- `dispatch_id`
- `report_id`
- `snapshot_id`
- `note_id`
- `world_pack_id`
- `faction_id`
- `base_id`
- `facility_id`

やること:

- 接頭辞付き ID を文字列型として整理する
- 将来的に branded type に上げられる余地を残す
- 少なくとも命名用の型 alias は分ける

### 5.2 列挙値の切り出し

対象:

- `member_kind`
- `role`
- `category`
- `kind`
- `phase`
- `availability`
- `mission_result`

やること:

- 仕様書内の列挙値を洗う
- 英小文字 `snake_case` に統一する
- UI 用表示名は別にし、内部型へ混ぜない

### 5.3 エンティティ型の定義

最低限必要な型:

- `WorldPack`
- `Faction`
- `Character`
- `StaffCharacter`
- `Base`
- `Facility`
- `Mission`
- `Dispatch`
- `Report`
- `Snapshot`
- `GuildmasterNote`

やること:

- JSON 叩き台の正準形をそのまま型へ落とす
- 任意項目と必須項目を分ける
- 1 フェーズ目では関数ロジックより型の明確さを優先する

### 5.4 相互参照の型表現

やること:

- 参照先は原則 `*_id` だけにする
- ネストオブジェクト内の `client` なども `faction_id` のように正準化する
- `relationships` や `linked_notes` などの配列構造も型へ落とす

### 5.5 型整合チェック

やること:

- サンプルオブジェクトを 1 件ずつ置く
- `tsc` で型が通ることを確認する
- 旧 `src/storage/types.ts` とはまだ統合しない

## 6. 実装時の注意

- 旧型定義へ直接書き足さない
- `mission` と `dispatch` と `report` を混ぜない
- `guildmaster_note` と `report` を同じ型へ押し込まない
- `obstacles` は必ず配列で持つ
- `world_pack_id` の値も正準形で揃える

## 7. 完了条件

- 全エンティティ型が存在する
- ID と列挙値の揺れがない
- JSON 叩き台の代表例を型で表現できる
- 型エラーなしでサンプルが通る
