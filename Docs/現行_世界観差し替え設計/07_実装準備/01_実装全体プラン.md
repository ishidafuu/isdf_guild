# 実装全体プラン

最終更新: 2026-03-10

この文書は、現行仕様を実装へ落とすための全体プランを整理する。
狙いは、仕様検討と実装作業を混線させず、最小縦切りから安全に前進することにある。

## 1. 前提

- 現行の正本仕様は `01_全体方針` から `06_確認事項` までに整理済み
- 最初に実装する対象世界観は `退廃サイバーパンク`
- 最初に通すデータ流れは `mission -> dispatch -> report -> snapshot -> guildmaster_note`
- 実装初期は「仕様の完全再現」ではなく、「1 周通ること」を優先する

## 2. 実装方針

- 既存の旧プロトタイプへ直接継ぎ足すのではなく、新仕様用の最小ルートを別系統で作る
- 型定義、サンプルデータ、シミュレーション、保存層を段階的にそろえる
- 永続化は最初から意識するが、最小縦切りの確認自体はメモリ上で成立してよい
- UI は後段に回し、最初は CLI / サンプル実行で確認できればよい

## 3. 実装フェーズ

### フェーズ 1. 型と正準データ構造の固定

目的:

- 仕様書で決めた JSON 叩き台を TypeScript 型へ落とす
- 命名規則と ID 規則をコード側へ持ち込む

対象:

- `world_pack`
- `faction`
- `character`
- `staff`
- `base`
- `facility`
- `mission`
- `dispatch`
- `report`
- `snapshot`
- `guildmaster_note`

完了条件:

- 型定義だけで各 JSON 叩き台を表現できる
- 主要 ID と参照関係がコード上でも一致している

### フェーズ 2. 最小実装用の実データ作成

目的:

- 退廃サイバーパンクで 1 周試せるデータ一式を用意する

対象:

- 世界観パック 1 件
- 勢力 3 件
- 冒険者 4 人から 5 人
- スタッフ 0 人または 1 人
- 拠点 1 件
- 設備 1 件から 2 件
- 依頼候補 3 件

完了条件:

- サンプルデータだけで次のフェーズへ進める
- 依頼選択肢が複数あり、人選の意味が出る

### フェーズ 3. 最小縦切りの通し実装

目的:

- 仕様の主ループを 1 回通す

流れ:

1. `mission` 候補の表示
2. `dispatch` の作成
3. `report` の生成
4. `snapshot` の生成
5. `guildmaster_note` 候補の生成と選択

完了条件:

- 1 依頼分の通し実行ができる
- `report` と `snapshot` が次回入力の土台になる
- `guildmaster_note` が人物側へ保存される

### フェーズ 4. 状態更新の実装

目的:

- 結果が次回へ残るようにする

更新対象:

- 人物の負傷
- 人物のストレス
- 人物同士 / 人物と勢力の関係
- 勢力との関係
- 拠点状態値

完了条件:

- 次回依頼時に前回の余波が反映される
- `snapshot` と実データの整合が取れている

### フェーズ 5. 永続化の導入

目的:

- 通し実行で生じた状態を保存し、再開できるようにする

対象:

- 主要データストア
- ID 単位の参照
- 更新トランザクション

完了条件:

- 依頼後の状態を保存し、再ロード後も継続できる

### フェーズ 6. AI 入出力の接続

目的:

- AI に任せる生成部分を実際の契約へ落とす

優先順位:

1. `report`
2. `guildmaster_note` 候補
3. `mission`
4. `dispatch` 補助

完了条件:

- `text` / `intent_tags` / `reason_summary` を含む出力が安定する
- 人物や状態を踏まえた結果文が破綻しない

### フェーズ 7. 試運転と調整

目的:

- 実際に数サイクル回して、不自然さや不足を確認する

確認したいこと:

- 成功 / 部分成功 / 失敗の流れが自然か
- 状態持ち越しが重すぎないか
- スタッフの口出しがうるさすぎないか
- `guildmaster_note` が負担にならないか
- 依頼の選択肢に意味が出るか

完了条件:

- 一通りの進行が確認できる
- 次の仕様修正が「実地の違和感」ベースになる

## 4. 実装の優先順位

1. 型定義
2. サンプルデータ
3. 最小縦切り
4. 状態更新
5. 永続化
6. AI 接続
7. 試運転

## 5. 現時点の完成条件

現時点で「実装前段階を抜けた」と言える条件は次の通り。

- `退廃サイバーパンク` の実データ一式がある
- `mission -> dispatch -> report -> snapshot -> guildmaster_note` が 1 本通る
- 前回の負傷や印象が次回へ反映される
- スタッフ、拠点、勢力が最低限関与している

## 6. 現時点で後回しにするもの

- 長期政治変動
- 複雑な拠点設備の中間状態
- 多世界をまたぐ同時運用
- UI の最終設計
- 複数人運用
