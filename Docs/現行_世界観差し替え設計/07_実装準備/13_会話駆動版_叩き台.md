# 会話駆動版 叩き台

最終更新: 2026-03-12

この文書は、最小実装を試運転した結果を踏まえ、
現行の `mission -> dispatch -> report -> snapshot -> guildmaster_note` を壊さずに、
表面上の体験を「依頼一覧から選ぶゲーム」ではなく「会話と相談で進むギルド運営ノベル」へ寄せるための叩き台である。

本書は正本の置き換えではなく、次回以降の調整用たたき台として扱う。

## 1. 現時点の違和感

最小 Web UI を動かした時点での主な違和感は以下である。

- ユーザー側として、いま何をすればよいかが見えにくい
- 文章が出ていても、読むこと自体が主役になっていない
- `mission` が最初から並んでいるため、依頼の生まれ方や相談の空気が薄い
- スタッフや冒険者とのやりとりが「補助」止まりで、場面を動かす主役になっていない
- プレイヤーの自由入力がまだ存在せず、会話ゲームとしての手触りが足りない

## 2. 目指す方向

### 2.1 体験の軸

このゲームの表面体験は、次の方向へ寄せる。

- プレイヤーは一覧を捌く管理者ではなく、ギルドの場にいる主として振る舞う
- 主な入力は「何を最適化するか」ではなく、「誰の話を聞き、どう返し、どう扱うか」である
- 依頼は最初から完成品として並ぶよりも、会話や相談の中で輪郭が見えてくる
- 数字は裏で回し、表では人物の反応、空気、口調差、含みのある返答を主に見せる
- `report` は事実、`guildmaster_note` は主観という分離は維持する

### 2.2 今回の仮説

今の違和感は「世界観が重い」ことよりも、
プレイの入口が会話ではなく一覧になっていることの影響が大きいとみなす。

したがって、次の改善軸は以下とする。

- 依頼を選ぶ前に、必ず誰かとのやりとりがある
- `mission` は会話の結果として確定する
- スタッフは案内役ではなく、口出し、観察、横やりの発生源として前に出す
- 冒険者側からの持ち込みや逆提案を入れる
- 自由入力は「発言そのもの」ではなく「意図入力」として扱う

## 3. 会話駆動版の 1 サイクル案

### 3.1 表面の流れ

表面上の 1 サイクルは、次の順で見せる。

1. 朝の場面
2. 話しかけてくる相手の発生
3. 相談 / 打診 / 持ち込み
4. プレイヤーの返答
5. 必要ならスタッフや他メンバーの口出し
6. 依頼条件の確定
7. 人選相談
8. 出発
9. 帰還
10. `report`
11. `guildmaster_note`

内部では引き続き、
`mission -> dispatch -> report -> snapshot -> guildmaster_note`
を主ループとして維持する。

つまり変更するのはコアループではなく、表面に出す順番と演出である。

### 3.2 入口候補

毎回の入口は 1 種類に固定しない。
最初は以下の 3 類型を持つとよい。

- スタッフの報告
  - 例: 「朝一番で面倒な客が来てる」「昨日の件で港湾側がまた話を持ってきた」
- 冒険者の持ち込み
  - 例: 「この案件、こっちで拾えそうだ」「あの依頼、自分を出してほしい」
- 依頼主の来訪
  - 例: 「この仕事を受けられるかだけでも聞きたい」

入口が違うだけで、同じ `mission` でも印象がかなり変わる。

## 4. `mission` の見せ方

### 4.1 現状の問題

今は `mission` が一覧で先に存在しているため、
依頼が「仕様上のカード」に見えやすい。

### 4.2 会話駆動版の扱い

会話駆動版では、`mission` を 2 段階で扱う。

- 内部:
  - 既存どおり canonical な `Mission` を持つ
- 表面:
  - まずは未確定の相談材料として提示する
  - 会話の結果、条件が詰まり、受注可否が見える
  - そのあとで `Mission` として確定した扱いに入る

必要なら後で `mission_seed` のような前段階データを追加する。
ただし最初は既存 `Mission` から会話文を逆算するだけでもよい。

## 5. プレイヤー入力の考え方

### 5.1 選択肢

基本は選択肢中心でよい。

例:

- 詳しく聞く
- いったん保留する
- 誰かの意見を聞く
- 今日は見送る
- その条件で受ける
- 報酬を確認する

### 5.2 自由入力

自由入力は全場面に広げず、以下から始めるのがよい。

- 質問
- 返答方針
- 指示の意図
- `guildmaster_note` の短い補足

### 5.3 自由入力の解釈方針

重要:
自由入力文は、そのままギルド主の発言として表示しない。

代わりに、AI は入力文を以下の順で扱う。

1. 入力文から意図を抽出する
2. その意図を、ギルド主の口調や行動に変換する
3. 相手に伝わる自然なやりとりとして再構成する

例:

- 入力:
  - 「危なそうだからその運び方はやめさせたい」
- AI の解釈:
  - 懸念表明 + 方法の変更要求
- 表示されるギルド主の行動:
  - 「その段取りは危ない。運ぶ時刻か経路を見直したい、と主は言った」

この方式なら、プレイヤーはチャット小説の台本を書く必要がなく、
「どういう意図で返したいか」だけに集中できる。

## 6. スタッフの位置づけ

### 6.1 基本役割

スタッフは数値補助ではなく、次の役割を担う。

- 案件の持ち込み
- 依頼主や冒険者の空気の翻訳
- 人選への口出し
- プレイヤー入力の補助解釈
- 話を進めるための横やり

### 6.2 出し方

スタッフは毎回必ず長く喋る必要はない。
ただし、会話駆動版では「場面を動かす人」として前面に出した方がよい。

最小実装では、まず 1 人を固定相談役として常駐させる。

### 6.3 スタッフの差

同じ懸念でも、スタッフによって言い方を変える。

- 率直
- 遠回し
- 面倒見がよい
- 疑い深い
- 実務的

これにより、攻略情報ではなく人物として機能させる。

## 7. 文章表示の優先順位

会話駆動版では、文章を次の優先順位で生成する。

1. いま何が起きているか
2. 誰が何を言っているか
3. どこに気まずさ、軽口、含みがあるか
4. 世界観らしい空気

気をつける点:

- 雰囲気が先に立って状況が分からなくならないこと
- 比喩を重ねすぎないこと
- 依頼内容、報酬、懸念、目的は読み取れるようにすること

## 8. 報酬の扱い

報酬額は出してよい。
むしろ依頼規模や危うさを伝えるために、分かりやすい指標として残した方がよい。

ただし、以下は前面に出しすぎない。

- 難度の数字
- ストレスの数字
- 露骨な成功率補正

これらは裏で使い、表では会話や所感に置き換える。

## 9. 最小 UI 叩き台

### 9.1 画面構成

1 画面に多数のカードを並べるより、以下の 3 領域程度がよい。

- 本文ログ
  - 地の文
  - 会話
  - 短い所感
- 選択肢 / 意図入力
  - 定型選択肢
  - 必要時のみ自由入力
- 記録欄
  - 今回の依頼名
  - 報酬
  - 前回メモ
  - 誰が今気になっているか

### 9.2 1 シーン 1 焦点

1 シーンでプレイヤーに見せる焦点は 1 つに絞る。

例:

- 依頼内容を聞く場面
- 誰に声をかけるか考える場面
- 帰還した空気を受け止める場面

複数の判断を 1 画面に同居させすぎない。

## 10. AI 生成の役割分担

### 10.1 AI に任せるもの

- 朝の導入
- 依頼の打診や相談の文面
- スタッフの返答
- 冒険者の持ち込みや逆提案
- プレイヤー入力の意図解釈
- 出発前会話
- 帰還後の短い場面
- `report` 本文
- `guildmaster_note` 候補

### 10.2 ルール側で固定するもの

- canonical な `Mission`
- `Dispatch`
- 判定結果
- `Snapshot`
- 参照 ID
- 進行順

文章の自由度は高くしてよいが、
事実関係の正準データは従来どおりルール側で固定する。

## 11. 次の実装順たたき台

最初から全面改修せず、次の順で小さく試すのがよい。

1. 依頼一覧の代わりに「入口シーン」を作る
2. スタッフ報告 / 冒険者持ち込み / 依頼主来訪の 3 パターンを足す
3. 選択肢に「誰の話を聞くか」「保留する」を足す
4. 自由入力欄を 1 箇所だけ入れる
5. 自由入力を意図解釈して返す AI 層を足す
6. `mission` 確定後に既存 `dispatch` へ接続する
7. 帰還後シーンを `report` 前に差し込む

## 12. まだ未確定の論点

現時点では、以下は決め切らずに試しながら調整する。

- プレイヤーの自由入力頻度はどれくらいが心地よいか
- 入力意図の解釈精度が低い時、どうフォローするか
- 依頼主、スタッフ、冒険者の誰を入口に一番多く使うか
- 1 サイクルのどこまでを会話にするか
- AI 生成文の長さをどこで切るか
- 相談中にどこまで内部情報を隠すか

## 13. 現時点の結論

現行の主ループ自体を捨てる必要はない。
ただし表面体験は、依頼カード選択型よりも、
「誰かが話を持ち込み、プレイヤーが聞き、返し、口を挟まれ、結果として依頼が固まる」
流れへ寄せた方が、いま目指しているゲーム像に近い。

自由入力については、
「プレイヤーが書いた文をそのまま喋らせる」のではなく、
「プレイヤーの意図を AI が汲み取り、ギルド主らしい返答や行動に変換する」
方式を基本線とする。
