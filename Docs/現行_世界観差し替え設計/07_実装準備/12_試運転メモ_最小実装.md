# 試運転メモ 最小実装

最終更新: 2026-03-10

この文書は、`退廃サイバーパンク` の最小実装を通した時点の確認メモである。

## 実行対象

- `src/examples/run_minimal_cyberpunk.ts`
- `src/examples/run_trial_cyberpunk.ts`

## 実行した確認

### 1. スタッフあり / 輸送寄り

- 対象依頼
  - `mission_0001`
- 期待したこと
  - 安全寄り依頼で成功系が自然に出るか
  - 大成功時でも `guildmaster_note` が 1 件以上残るか
- 確認結果
  - 大成功を確認
  - `guildmaster_note` が 1 件保存されることを確認
  - 状態持ち越しは軽微で、次回依頼に進める

### 2. スタッフあり / 交渉寄り

- 対象依頼
  - `mission_0002`
- 期待したこと
  - `negotiator` と `scout` の意味が出るか
  - スタッフのコメントが人選補助として機能するか
- 確認結果
  - 成功を確認
  - 交渉役中心の編成が自然に選ばれる
  - 依頼後メモも 1 件以上残る

### 3. スタッフなし / 危険寄り

- 対象依頼
  - `mission_0003`
- 期待したこと
  - 失敗時の損耗と持ち越しが見えるか
  - スタッフなしでも主ループが止まらないか
- 確認結果
  - 失敗を確認
  - `resting` / `unavailable` への遷移を確認
  - スタッフなしでも `mission -> dispatch -> report -> snapshot -> guildmaster_note` は通る

### 4. 永続化

- 保存先
  - `/tmp/isdf_guild/cyberpunk_minimal_state.json`
- 確認結果
  - 1 サイクル後に保存
  - 再読込後に 2 サイクル目を継続実行できることを確認

## よかった点

- `mission` と `dispatch`、`report` と `guildmaster_note` の責務分離は保てている
- `snapshot` を `post_report` で切っており、次回への現在地として使いやすい
- 5 人の冒険者に役割差と性格差があり、依頼ごとの人選差が出る
- スタッフあり / なしの両方を最小コストで比較できる

## 現時点の違和感

- 成功系の `guildmaster_note` は今は簡易的で、まだ人物の深い主観差までは弱い
- `report.text` はテンプレート生成なので、世界観の乾いた文体は出ているが変化幅は狭い
- 関係更新は最小限で、人物同士の相性が強く効く段階にはまだ達していない
- `storage_v2` は JSON ファイル保存で、更新競合や履歴管理までは未対応

## 残る不明点

- `guildmaster_note` の `source_kind` は将来的に `rest_phase` や手動入力を含めるか
- `snapshot.faction_states[].relation_state` をどこまで列挙値で固定するか
- `facility` の効果を今後どこまでロジックへ反映するか
- `mission` の生成を固定データから AI 生成へ切り替える際、危険度と依頼数のバランスをどう保つか
- 永続化を今後も単一 JSON で続けるか、ストア分割を進めるか

## プレイ後に決めるべきもの

以下は、現時点では無理に先決めせず、実際に数サイクル回してから判断した方がよい論点である。

- スタッフごとの個性の強さ
  - どこまで観察力、口出し頻度、言い方の差を強めるか
- 火種の種類の追加範囲
  - 現在の最小セットで十分か、種類を増やした方が読み味がよくなるか
- 成功時余波の濃さ
  - 成功でもどれくらい後味や違和感を残すか
- 火種の残りやすさ
  - 人物性格による残留差が強すぎないか、弱すぎないか
- 関係変化の見え方
  - 会話やメモへの滲み方が読み取りやすいか、逆に説明不足すぎないか
- スタッフなし時の情報量
  - 冒険者会話だけで十分か、もう少し読み取り材料が要るか
- `report` の具体性
  - 事実中心のままいくか、場面描写や誰が何をしたかを増やすか
- `storage_v2` の強化優先度
  - 長期試運転前に保存層を強めるか、このまま挙動確認を優先するか

## プレイ中に注視すべき点

以下は、自由記述感想を書くときに意識して見ると判断材料になりやすい観点である。

### 1. 編成時の迷い

- 誰を出すかで本当に少し悩めるか
- 能力だけでなく、相性や消耗、因縁を自然に気にしたくなるか
- 「最適解を探す」より「なんとなく嫌な感じがする」が発生しているか

### 2. スタッフの台詞

- ただの説明文ではなく、そのスタッフらしい物言いに見えるか
- 直接言いすぎていないか
- 逆に曖昧すぎて何も拾えない感じになっていないか

### 3. スタッフなし時の読み味

- 説明が減ることで、逆に想像の余地が出ているか
- 冒険者会話や態度から何かを感じ取れるか
- 情報不足で単に分かりにくいだけになっていないか

### 4. 成功時の後味

- 成功でも「何かが少し残った」感じがあるか
- それが気持ちよさを壊しすぎていないか
- 逆に成功があっさり流れすぎていないか

### 5. `guildmaster_note`

- ただの結果要約ではなく、その人物をどう見たかになっているか
- 成功時メモでも人物差が少し感じられるか
- 同じ人物に毎回似たメモばかり出ていないか

### 6. 次回への余波

- 前回のことが、次の人選や気分に少しでも残っているか
- 何がどう影響したかを完全に説明されなくても、何となく筋が通っているか
- 読み取れた時に面白さがあるか

### 7. 読み取れなかった時の納得感

- うまく読めなかった結果として変なことが起きても、それが物語として許容できるか
- プレイヤーの誤読や見落としが、単なる理不尽ではなく「そういうこともある」に収まるか

### 8. 作業感の有無

- 依頼を1本片付けた時に、単なるルーチンに感じるかどうか
- 次の依頼に行く前に「少し考えたい」と思えるか
- その考えたくなる理由が、数字ではなく人物や空気に寄っているか

## AI文章生成の待ち時間短縮アイデア

以下は、後で導入する可能性がある前提での検討メモである。
現時点では未実装だが、CLI 連携や将来の API 接続へそのまま流用しやすい順で整理する。

### 1. 読んでいる間の先読み生成

- `briefing` を表示した直後に、裏で `casting` 用シーン文を先に生成する
- 人選が確定した直後に、裏で `aftermath` の場面文候補を先に投げる
- `aftermath` 表示中に、次の `briefing` 候補を生成し始める
- `guildmaster_note` 候補を表示した時点で、次の依頼紹介文や脇案件紹介文を生成する

狙い:

- プレイヤーが文章を読んでいる時間を、そのまま次段階の生成時間に変える
- 画面遷移時の待機を減らす

注意:

- 人選や結果で内容が変わる箇所は、先読みが外れた時に破棄される前提が必要
- CLI の同時起動数を制限しないと逆に重くなる

### 2. 2段階生成

- 先に短い暫定文だけ返す
- その後で長めの本命文へ差し替える

例:

- `briefing`
  - まず 1 行だけ
  - 後で地の文 3 行に拡張
- `aftermath`
  - まず 1 段落だけ
  - 後で人物別の帰還反応を追加

狙い:

- 完全な待ち時間を減らす
- プレイヤーに「止まっている」印象を与えにくくする

注意:

- 差し替え時に読んでいる最中の文面が変わると違和感がある
- UI 上は「追記」寄りに見せる方がよい

### 3. 粒度分割

- 1回の大きい生成ではなく、以下を分けて生成する
  - 地の文
  - スタッフ台詞
  - 人物反応
  - 脇の記録文

狙い:

- 速く返せる部分から先に出せる
- 失敗しても一部だけ fallback しやすい
- キャッシュ単位を細かくできる

注意:

- 生成回数が増えすぎると CLI では逆効果になりやすい
- API 移行後の方が相性は良い

### 4. キャッシュの強化

- すでに入っている `prompt + schema` 単位キャッシュに加え、以下の粒度を検討する
  - `mission_id + stage`
  - `mission_id + selected_character_ids`
  - `report_id`
  - `character_id + report_id`
- TTL を持たせる
- 明示的なキャッシュ削除導線を作る
- 開発時は `fallback` と `CLI` のどちらが返ったかを見えるようにする

狙い:

- 同じ場面の再読で待たない
- リロードや試運転中の体感をかなり改善する

注意:

- 火種や人物状態が少しでも違うと別キャッシュにすべき箇所がある
- 雑に再利用すると「古い空気」が出る

### 5. 投機生成

- いまの選択から「次に選ばれそうな候補」を予測して生成しておく

例:

- `briefing` 中に、最初の依頼だけでなく次点依頼の紹介文も生成
- `casting` 中に、よくある人選パターンを何通りか先に生成
- `aftermath` 中に、次サイクルの最初の朝文を生成

狙い:

- プレイヤーの選択が当たれば待ち時間がほぼ消える

注意:

- 外れた生成コストは無駄になる
- CLI 連携ではやりすぎると遅くなる

### 6. 生成キュー

- 生成要求をその場で全部投げず、優先順位付きキューで処理する

優先順位案:

1. いま画面に必要な本文
2. 人物反応
3. 脇の記録や補足
4. 次画面の先読み

狙い:

- 体感に効く文から先に返す
- 並列数を制御しやすい

注意:

- 今後 API に移るなら本格導入しやすい
- CLI 試作段階では簡易キューでも十分

### 7. 画面遷移単位のプリフェッチ

- 画面を開いた瞬間に次の画面で必要なデータ生成を始める

例:

- `briefing` を開いた瞬間
  - `casting` の地の文
  - 依頼人説明
- `casting` を開いた瞬間
  - 選択中メンバーの反応
  - 想定 `aftermath`
- `aftermath` を開いた瞬間
  - 次の `briefing`
  - 脇案件紹介

狙い:

- プレイヤーが読む時間を最大限使える

### 8. 人物反応だけ先にキャッシュ生成

- 人物ごとの反応文は変動幅が比較的小さいため、依頼カテゴリ単位で先に作っておく

例:

- `char_shion + delivery`
- `char_mina + recovery`
- `char_gai + negotiation`

狙い:

- 人選画面の反応表示をかなり速くできる
- 後で火種や当日の空気だけを追記する構成にもできる

注意:

- 固定化しすぎると毎回同じ文になりやすい
- 変化分だけ別生成する構造が必要

### 9. report と scene の役割分担見直し

- `report` は保存用の事実文に寄せる
- 読ませる本文は scene 側で厚くする

狙い:

- 保存文を短く固定しやすい
- scene の先読みや差し替えがしやすくなる
- 待機時間を scene のみに集中させられる

### 10. 今後の導入優先度案

短期で効く:

- `briefing` 表示中の `casting` 先読み
- `aftermath` 表示中の次 `briefing` 先読み
- 生成キュー
- キャッシュキー強化

中期で効く:

- 2段階生成
- 粒度分割
- 人物反応の事前生成

CLI 試作段階では重そう:

- 投機生成の多パターン展開
- 細粒度並列生成の多重化

### 11. 後で判断したい観点

- 先読みが当たる率はどの程度か
- キャッシュで文章の新鮮さが落ちていないか
- 「待つ時間」より「差し替わる違和感」の方が気になるか
- CLI でどの程度まで並列化して耐えられるか
- API 移行後に、どこまで段階生成を広げるか
