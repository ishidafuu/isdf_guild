# フェーズ3 最小縦切りの通し実装

最終更新: 2026-03-10

このフェーズでは、新仕様の主ループを 1 回通す。
ここでいう主ループは `mission -> dispatch -> report -> snapshot -> guildmaster_note` である。

## 1. このフェーズの目的

- 実装全体の中核フローを先に成立させる
- 各データが単独で終わらず、連動することを確認する
- UI なしでもゲームの骨格が動く状態を作る

## 2. 入力

- フェーズ1 の型定義
- フェーズ2 の固定データ

## 3. このフェーズで作るもの

- 依頼候補取得処理
- `dispatch` 作成処理
- `report` 生成処理
- `snapshot` 生成処理
- `guildmaster_note` 候補生成処理
- 通し実行用サンプルスクリプト

## 4. 推奨ディレクトリ

- `src/core/mission_flow/`
- `src/core/note_generation/`
- `src/examples/`

推奨ファイル:

- `selectMission.ts`
- `buildDispatch.ts`
- `buildReport.ts`
- `buildSnapshot.ts`
- `buildNoteCandidates.ts`
- `runMinimalMissionCycle.ts`
- `run_minimal_cyberpunk.ts`

## 5. 作業項目

### 5.1 依頼候補表示

やること:

- 依頼候補 3 件を返す
- 依頼の表示用情報を整える
- ここでは AI 生成ではなく固定データでよい

### 5.2 `dispatch` の作成

やること:

- 選ばれた依頼を参照する
- 参加人物を決める
- 懸念や注記を持つ
- 使用設備を選ぶ

最低限持たせるもの:

- `mission_id`
- `assigned_character_ids`
- `base_state.selected_facility_ids`
- `concerns`
- `assignment_reason_summary`

### 5.3 `report` の生成

やること:

- 依頼結果を確定する
- `text`
- `intent_tags`
- `reason_summary`
- `state_updates`
を返す

この段階では:

- 完全 AI 生成でなくてもよい
- テンプレート生成でも可

### 5.4 `snapshot` の生成

やること:

- `post_report` で切る
- 人物状態
- 勢力状態
- 拠点状態
をまとめる

### 5.5 `guildmaster_note` 候補の生成

やること:

- AI 風の仮メモ候補を 2 件から 4 件ほど出す
- 1 件以上選べる構造にする
- 短文補足を保存できる形にする

### 5.6 サンプル実行

やること:

- 1 本のサンプルスクリプトで全工程を出力する
- ログとして追えることを優先する

## 6. 実装時の注意

- まずは「綺麗さ」より「1 本通ること」
- `dispatch` と `report` を混ぜない
- `guildmaster_note` は `report` に埋め込まない
- `snapshot` は `report` の後に切る

## 7. 完了条件

- 依頼選択からメモ保存まで通る
- 途中の参照 ID がすべてつながる
- 少なくとも 1 パターンの通し確認ができる

## 8. 次フェーズへの引き継ぎ

次フェーズでは、ここで生成された `report` と `snapshot` を使って状態更新を厳密化する。
この時点で無理に永続化まで持ち込まなくてよい。
