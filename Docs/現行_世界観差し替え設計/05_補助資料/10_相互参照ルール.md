# 相互参照ルール

最終更新: 2026-03-10

この文書は、人物、依頼、日報、勢力、世界観パックの JSON 叩き台をまたいで、
どのデータがどの ID を参照し、どの方向に接続するかを整理する。

補足:
命名規則や ID 規則は `07_JSON命名規則とID規則.md` に従う。

## 1. 目的

- JSON ごとの孤立を防ぐ
- 実装時に参照方向を迷わないようにする
- 循環参照を必要以上に増やさない
- 最小実装に必要な接続だけを先に固定する

## 2. 基本方針

- 参照は原則 `*_id` で持つ
- 詳細データを重複保持しない
- 表示用の名前は必要最小限だけ冗長に持ってよい
- 1 つの事象は 1 つの主データを持ち、他はそこを参照する

例:

- 依頼の結果は `日報`
- 人物の印象は `guildmaster_note_log`
- 世界観固有の語彙は `世界観パック`
- 勢力の立場は `勢力データ`

## 3. 主データごとの参照関係

### 3.1 人物データ

人物データが持つ主な参照:

- `world_pack_id`
  - 所属する世界観パック
- `relationships[].target_id`
  - 他キャラまたは勢力
- `guildmaster_note_log[].source_id`
  - 日報や面談などの発生源

人物データを参照する主な側:

- 日報
  - `fact_log.injury_targets`
  - `fact_log.stress_targets`
  - `state_updates.character_updates[].character_id`
  - `linked_notes[].character_id`
- 依頼
  - `participants.recommended_roles` は直接の人物参照ではなくロール参照に留める

### 3.2 依頼データ

依頼データが持つ主な参照:

- `world_pack_id`
  - 所属する世界観パック
- `client.id` または `client.faction_id`
  - 依頼人が勢力なら勢力データ

依頼データを参照する主な側:

- 日報
  - `mission_id`
- 人物メモ
  - 間接的に `source_id = report_id` 経由でつながる

### 3.3 日報データ

日報データが持つ主な参照:

- `mission_id`
  - 対象依頼
- `world_pack_id`
  - 世界観パック
- `fact_log.relation_changes[].target_id`
  - 勢力または人物
- `state_updates.character_updates[].character_id`
  - 人物
- `state_updates.faction_updates[].faction_id`
  - 勢力
- `linked_notes[].character_id`
  - 人物
- `linked_notes[].note_candidate_set_id`
  - メモ候補セット

日報データを参照する主な側:

- 人物データ
  - `guildmaster_note_log[].source_id`

### 3.4 勢力データ

勢力データが持つ主な参照:

- `world_pack_id`
  - 所属する世界観パック
- `conflicts[].target_faction_id`
  - 他勢力

勢力データを参照する主な側:

- 依頼
  - `client`
- 人物
  - `relationships[].target_id`
- 日報
  - `relation_changes[].target_id`
  - `faction_updates[].faction_id`
- 世界観パック
  - `faction_ids`

### 3.5 世界観パックデータ

世界観パックが持つ主な参照:

- `faction_ids`
  - 勢力群

世界観パックを参照する主な側:

- 人物
  - `world_pack_id`
- 依頼
  - `world_pack_id`
- 日報
  - `world_pack_id`
- 勢力
  - `world_pack_id`

## 4. 参照方向の推奨

### 推奨する方向

- 世界観パック → 勢力
- 依頼 → 世界観パック
- 依頼 → 勢力
- 日報 → 依頼
- 日報 → 人物
- 日報 → 勢力
- 人物 → 世界観パック
- 人物 → 日報
- 人物 → 他人物 / 勢力

### 避けたい方向

- 依頼が人物を直接固定列で抱え込む
- 世界観パックが人物や依頼の個別 ID を直接持つ
- 勢力が人物一覧を常時抱える

補足:
人物参加は「依頼が固定メンバーを持つ」のではなく、
セッション時のアサイン結果として別データへ切り出す方が拡張しやすい。

## 5. 最低限そろえるべき相互参照

最小実装で最低限そろえるべきもの:

- 人物 `world_pack_id`
- 依頼 `world_pack_id`
- 勢力 `world_pack_id`
- 日報 `mission_id`
- 日報 `state_updates.character_updates[].character_id`
- 日報 `state_updates.faction_updates[].faction_id`
- 人物 `guildmaster_note_log[].source_id`

これだけで、
「どの世界の、どの依頼で、誰に何が起きたか」
までは一通りつながる。

## 6. 後から足してよい接続

- `note_candidate_set_id`
  - AI 下書き候補セット
- `timeline` と `report_id` の接続
- 勢力同士の詳細対立ネットワーク
- 拠点設備と人物休養イベントの参照

## 7. 実装上の注意

- 詳細本文より ID 接続を先に安定させる
- 相互参照は増やしすぎず、日報を中心ノードにすると整理しやすい
- プレイヤー主観は `guildmaster_note_log`、事実は `日報` に分ける
- 世界観ごとの違いは `world_pack_id` から引く
