# framework 内部レビュー検討メモ

最終更新: 2026-03-10

## 1. 目的

本章は、追加した `framework` 文書群に対して、異なる観点の内部レビュアーを仮定して論点を洗い出し、現時点の採用方針を残すためのメモである。

## 2. レビュアー観点

### 2.1 システム設計レビュアー

見る点:

- 世界観差し替え時に処理が壊れないか
- データ構造が実装しやすいか
- 共通機能名が肥大化していないか

主なコメント:

- 能力値5種、ロール5種は妥当
- 依頼カテゴリ8種は現状十分だが、採取カテゴリを独立させるかは後で再確認が必要
- 世界観辞書は「表示名辞書」と「候補語辞書」を分けてもよい

現時点の採用方針:

- 依頼カテゴリは8種のまま維持
- 採取は当面 `recovery` へ含める
- 辞書はまず人間が読める形で整備し、実装時に JSON 化を検討する

### 2.2 ゲーム体験レビュアー

見る点:

- 世界観を変えたときに本当に遊び味が変わるか
- キャラ差が判定以外でも感じられるか
- 成長の楽しみが十分にあるか

主なコメント:

- 用語差し替えだけでは弱いので、報酬価値と損害表現の差は維持すべき
- キャラ差は特技と欠点の噛み合わせが重要
- 成長は能力値よりも特技と関係値の変化を主役にしたほうがギルドゲーム向き

現時点の採用方針:

- 世界観辞書では報酬、障害、損害表現を厚めに持つ
- キャラ設計では欠点と性格タグを残す
- 成長は特技中心で検討する

### 2.3 生成運用レビュアー

見る点:

- 自動生成や半自動生成に落とし込みやすいか
- 生成結果の品質管理がしやすいか
- 辞書の差し替えが機械処理に向くか

主なコメント:

- 依頼生成表は十分機械化しやすい
- ただし辞書候補に粒度の差があるため、将来的には「依頼人」「場所」「脅威」「報酬」ごとに件数を揃えた方が良い
- キャラ特技も機能タグを付けると扱いやすい

現時点の採用方針:

- 次段階で辞書エントリに機能タグを足す余地を残す
- 特技テンプレに `有効カテゴリ` を含めた
- 品質チェックは文章よりも構造先行で行う

### 2.4 物語運用レビュアー

見る点:

- 依頼が毎回ただの作業にならないか
- キャラ欠点が物語に効くか
- 世界観ごとの情緒が消えていないか

主なコメント:

- 共通構造が強いぶん、依頼ごとの「隠し事」や「余波」が重要
- 和風怪異は説明しすぎると雰囲気が飛ぶので、辞書側でも曖昧さを維持する
- サイバーパンクは報酬と契約条件で冷たさを出すべき

現時点の採用方針:

- 追加事情表を毎回1つ乗せる運用を推奨
- 世界観辞書にはトーンガイドを付けた
- 欠点は単なる数値デメリットではなく、事件の火種として使う想定を維持する

## 3. 今後の改善候補

- 特技を機能タグ付き一覧にする
- 世界観辞書を JSON 的構造に再整理する
- キャラ出自辞書を世界観別に追加する
- 装備カテゴリの共通仕様を定義する
- 関係値の変化ルールを独立章にする
- 数字を追いすぎない軽量ギルド運営モデルを独立章にする

## 3.1 軽量ギルド運営モデルの提案

現時点の方針では、ギルド運営は案件と人材運用の背景として機能するべきであり、数字を強く追う主ゲームにはしない方がよい。

おすすめの軸は以下。

- `状態ベース運営`
  - 金額や生産量を細かく積まない
  - 代わりに「資金難」「人手不足」「評判上昇」「派閥警戒」などの状態で管理する
- `設備ベース運営`
  - 施設を数値レベルではなく、設備の有無で管理する
  - 例: 医務室がある、記録庫がある、交渉室がある、工房がある
- `案件開放ベース運営`
  - ギルド成長は、受けられる依頼の幅や紹介される人物の層が増える形で見せる
- `人物運用ベース運営`
  - 誰を休ませるか、誰を組ませるか、誰を危険依頼に出すかを主な経営判断にする

特に相性がいいのは、以下の4指標を「大雑把な状態」として持つ方式。

- 評判
  - 依頼の質や紹介先に影響
- 人手余力
  - 連続依頼にどこまで対応できるか
- 設備充実度
  - 対応できる依頼カテゴリや回復速度に影響
- 外部関係
  - 派閥、依頼人、地域との関係性

この方式なら、数字を追う経営ではなく「ギルドの空気と体制が変わっていく」感覚を出しやすい。

## 3.2 現時点の推奨

レビュー観点からの推奨は以下。

- 財政を細かい数値会計にしない
- 施設は有無と効果だけで管理する
- 人員管理は疲労、負傷、関係性を主軸にする
- 派閥関係は数値よりも状態やタグで扱う
- ギルド成長は「できることが増える」方向で見せる

## 4. 総評

現状の `framework` は、企画検討段階としては十分に一貫している。
次に仕様密度を上げるべき箇所は以下の3つ。

1. キャラ成長ループの具体化
2. 世界観辞書のデータ構造化
3. 依頼生成結果をシナリオ1本に束ねるルール
