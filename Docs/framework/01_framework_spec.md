# 世界観差し替え型ギルドゲーム 仕様草案

最終更新: 2026-03-11

## 1. 基本方針

本ゲームは、ギルド運営と依頼解決をコア体験とする。
世界観はコアルールの上に載る差し替えレイヤーとして扱い、ゲームの進行構造そのものは変えない。
各世界観は完全に別世界として扱い、多元世界の別地域として無理に接続しない。

本仕様では以下を目的とする。

- メインループを固定する
- 世界観ごとの差分をモジュール化する
- 人力制作と自動生成の両方に対応する
- 世界観追加時に既存ルールを壊さない

## 2. ゲームのコア構造

以下の要素は全世界観で共通とし、変更しない。

- プレイヤーはギルド所属キャラクターを運用する
- ギルドは依頼を受注し、達成によって資源を得る
- 資源を用いてキャラクターおよびギルドを成長させる
- セッションは依頼単位で進行する
- 成否に応じて報酬・損害・世界変化が発生する

補足:
本作は、物語性の高い経営ゲームを目指す。
成長の主役はキャラクター同士の物語と関係変化に置き、ギルド拡張は任意の成長軸として扱う。
プレイヤーが拡張したければ大きくでき、したくなければ現状維持寄りでも遊べる設計を目指す。
個人成長とギルド成長のどちらか一方を強く押し出すよりも、両方がゆるやかに変化していく状態を基本とする。
王道ファンタジーは基準世界ではなく、初期候補群のうちの並列な1世界観として扱う。
ギルド規模は「大きくすれば正解」にはせず、大きくなったなりの利点と重さが両方増える設計を目指す。
街や組織の変化は、数字の上昇そのものよりも、空気、立場、人間関係、受けられる仕事の変化として感じられる形を優先する。

## 3. メインループ

全世界観で共通のゲーム進行は以下とする。

1. 依頼提示
2. 依頼選択
3. 準備
4. 任務遂行
5. 判定解決
6. 結果反映
7. 報酬獲得
8. 成長・拠点更新
9. 次回依頼の解放

補足:
各世界観は、この流れの名称や演出を変更してよいが、段階の機能は変更してはならない。

## 4. コアルール固定領域

以下は世界観による変更不可領域とする。

- 判定方式
- 難易度の段階構造
- 成功 / 失敗 / 大成功 / 部分成功などの結果区分
- キャラクター成長の基本枠
- ギルド成長の基本枠
- 依頼カテゴリの機能分類
- リソース管理の基礎構造
- 失敗時の処理原則
- セッション進行順

## 4.1 現時点の固定方針

以下は、2026-03-11 時点で暫定的に固定する方針である。

- 判定方式は `2D6` とする
- 判定段階は `大成功 / 成功 / 部分成功 / 失敗` の4段階とする
- 戦闘は原則として「依頼中の1障害」として軽く扱う
- 1セッションの基本単位は軽めの短編依頼1本とする
- 短編依頼1本の想定時間は 15 分から 30 分程度とする
- 依頼1本の標準構成は 3 場面程度とする
- 死亡や引退などの恒久損失は、固定ルールではなくゲーム開始時の設定項目として切り替えられるようにする
- 負傷やストレスは、原則としてセッションをまたいで残るものとする

## 4.2 キャンペーン設定項目

世界観パックとは別に、ゲーム開始時にキャンペーン全体の温度感を決める設定項目を持たせる。

例:

- 恒久損失の有無
  - 死亡あり / 引退のみ / 恒久離脱なし
- 負傷の重さ
  - 軽め / 標準 / 重め
- 依頼失敗の重さ
  - 軽め / 標準 / 重め

補足:
これにより、同じコアルールでも「人が簡単には死なない群像劇」と「人が欠けていく苦い運用劇」の両方を扱えるようにする。

## 4.3 ギルド規模の考え方

ギルド規模の拡大は、単純な上位互換として扱わない。
規模が大きくなるほど、受けられる仕事や影響力が増える一方で、抱える責任や内部問題も増えるようにする。

規模拡大で増えるものの例:

- 依頼数の増加
- 上位案件や特殊案件への接続
- 派閥や権力との接点
- 抱える人員と設備の幅

規模拡大で増える負担の例:

- 内部の不和や人間関係のもつれ
- 外部からの期待や干渉
- 守るべき評判の重さ
- 人手不足や管理負担の深刻化

補足:
理想は、ユーザーが「自分にとって心地よい規模」を探っていけることにある。

## 5. 共通データ構造

世界観差し替えに対応するため、内部的には機能名で管理する。

### 5.1 能力値

能力値名は世界観ごとに表示変更可だが、内部機能は固定とする。

- `power`
  - 武力、制圧、突破
- `tech`
  - 工作、修理、分析
- `sense`
  - 察知、追跡、探索
- `social`
  - 交渉、説得、取引
- `will`
  - 精神抵抗、集中、継戦

### 5.2 ロール機能

- `frontliner`
  - 前線役。脅威を受け止め、排除し、突破口を作る役割
- `support`
  - 支援役。回復、強化、防護、継戦補助を担う役割
- `scout`
  - 探索役。索敵、追跡、先行確認、異常発見を担う役割
- `engineer`
  - 技術役。修理、解除、分析、特殊装置対応を担う役割
- `negotiator`
  - 渉外役。交渉、説得、聞き込み、利害調整を担う役割

表示名は世界観ごとに変更可能。

### 5.3 依頼カテゴリ

以下は共通カテゴリとして保持する。

- `subjugation`
  - 討伐。脅威を排除、無力化、鎮静化する依頼
- `escort`
  - 護衛。人物や物資を安全に守る依頼
- `investigation`
  - 調査。原因、真相、所在、証拠を明らかにする依頼
- `recovery`
  - 回収。物品、情報、人材、遺物などを確保する依頼
- `delivery`
  - 輸送。対象物を期限内に運ぶ依頼
- `negotiation`
  - 交渉。対立相手との合意や取引成立を目指す依頼
- `defense`
  - 防衛。拠点や人物を一定時間守りきる依頼
- `rescue`
  - 救助。危機にある対象を発見し保護する依頼

各世界観は表示文言を差し替えてよい。

### 5.4 基本リソース

- `hp`
  - 体力。負傷や損耗の蓄積を表す
- `supply`
  - 物資。消耗品、補給、任務継続資源を表す
- `money`
  - 資金。金銭や等価資源を表す
- `reputation`
  - 評判。信用、名声、組織評価を表す
- `action` または `fatigue`
  - 行動余力または疲労。1任務中の継続力を表す

表示名や演出名のみ変更可能。

## 6. 世界観パックの責務

世界観パックは以下を定義するモジュールとする。

- 世界の時代感
- ギルドの立場
- 主な脅威
- 勢力一覧
- 拠点の外観
- 依頼カテゴリの表示名
- ロール名の表示名
- 報酬の名称
- 消耗品や装備の名称
- ランダムイベント表
- 敵カテゴリ表
- 導入文と文体
- 固有フレーバー要素

補足:
世界観パックは、ルール処理ではなく表示・解釈・演出を担当する。
ただし、同じコアループを保つ範囲で、依頼の手触りや判断の重みなどのプレイ感に差が出ることは許容する。

## 7. 世界観が変更してよい範囲

以下は世界観差分として自由に変更可能。

- 用語
- テキスト文体
- 依頼文
- 敵の見た目と分類名称
- 勢力名称
- 通貨や報酬の呼称
- 拠点名称
- 特殊イベント表
- 状態異常や損害の演出名
- 特技や装備のフレーバー記述

## 8. 世界観が変更してはいけない範囲

以下を変更するとゲーム互換性が壊れるため禁止とする。

- 判定の根本式
- 成長曲線の基本設計
- 依頼1件あたりの進行構造
- ロール機能の本質
- 難易度と報酬の対応原則
- 失敗時の代償発生構造
- ギルド成長の主要ループ

## 9. 世界観パックの最小構成

1つの世界観パックは、最低限以下を含める。

- パック名
- 一文説明
- 世界説明 3行
- ギルド説明 3行
- 勢力 3つ以上
- 依頼カテゴリ言い換え一覧
- ロール名言い換え一覧
- 敵カテゴリ 8件以上
- イベント表 6件以上
- 報酬名称
- 拠点名称
- 導入シナリオ1本分の種

補足:
初期ワールドの本数は、先に固定しない。
まずは代表的なパックをいくつか用意して実際に動かし、別の世界観も十分に回せると判断できた段階で追加していく。

推奨の初期試験パック:

- 王道ファンタジー
- 退廃サイバーパンク
- 和風怪異

## 9.1 世界観差の許容範囲

世界観差は、見た目や語彙だけに限定しない。
同じコアループを維持する限り、以下の差分は許容する。

- 依頼で重視される障害の種類
- 報酬の価値観
- 損害や余波の重み
- 交渉と戦闘の比重
- 情報秘匿や因習の濃さ

補足:
ただし、差分が大きすぎて別ゲーム化しないよう、判定方式と依頼処理構造は固定する。

## 10. 世界観生成テンプレート

自動生成または半自動生成用に、以下の入力項目を定義する。

- `era`
  - 中世 / 近代 / 現代 / 未来 / 終末後 など
- `scale`
  - 都市 / 地方 / 国家 / 宇宙圏 など
- `guild_role`
  - 請負組織 / 互助会 / 調査機関 / 交易団 など
- `threat`
  - 魔物 / 怪異 / 企業 / 機械 / 災害 / 海賊 など
- `mission_tone`
  - 戦闘寄り / 探索寄り / 交渉寄り / 輸送寄り
- `reward_value`
  - 金 / 物資 / 信頼 / 名声 / 権利 / 情報
- `base_type`
  - 酒場 / アジト / 社務所 / 船 / 宇宙港 / 学園部室
- `mood_tags`
  - 王道 / 怪奇 / 退廃 / 青春 / 哀愁 / 開拓
- `taboo`
  - その世界で出したくない要素
- `unique_hook`
  - その世界固有の1要素

## 11. 世界観生成ルール

生成時は以下の原則に従う。

- コアループに合わない設定は採用しない
- 依頼カテゴリ8種に自然に変換できること
- ギルドという組織形態が成立していること
- 報酬概念が成立していること
- 拠点成長の余地があること
- 主要脅威が依頼化可能であること

## 12. 品質チェック条件

新規世界観パックは以下を満たす場合のみ採用する。

- 依頼カテゴリ8種のうち6種以上に自然な名称を与えられる
- ロール機能5種すべてに対応する役割名を与えられる
- 報酬、敵、勢力、拠点がそれぞれ明確に定義されている
- 任務失敗時の不利益を世界観上自然に説明できる
- 既存世界観と十分に雰囲気差がある
- 1セッション分の導入を即座に作れる

## 13. 実装上の推奨設計

データ管理は以下の分離を推奨する。

- `core`
  - 判定、進行、成長、依頼処理
- `world_pack`
  - 用語、勢力、敵表、依頼文、イベント表
- `scenario`
  - 個別シナリオ、地域差分、NPC配置

## 13.1 判定処理のたたき台

判定のたたき台は以下とする。

- 基本式:
  - `2D6 + 能力値 + 補正 >= 目標値`
- 判定結果:
  - 目標値より 2 以上高い: 大成功
  - 目標値以上: 成功
  - 目標値より 1 低い: 部分成功
  - 目標値より 2 以上低い: 失敗

補足:
部分成功は「目的を一部達成するが、代償や不足が残る」結果として扱う。

## 13.2 依頼生成運用方針

依頼カテゴリは、現時点では 8 種で維持する。
採取や潜入の独立カテゴリ化は、実際の試運転後に必要性を見て再判断する。

また、依頼生成は最終的に完全ランダムでも成立するよう設計する。
ただし、生成結果の品質確認や手直しを妨げるものではない。

依頼内容の方針:

- 毎回必ず隠し事や追加事情を入れる必要はない
- 軽い日常依頼も許容する
- 重い依頼と日常依頼の両方が混ざることで、ギルド運営らしい流れを作る
- 失敗時の余波の重さは、依頼内容と関係者に応じて変える
- 依頼人や勢力との関係により、案件内容や条件が変質する構造は強めに持たせてよい

## 14. 設計思想の要約

本ゲームは「世界観固定の作品」ではなく、
「ギルド運営ゲームのコアシステムに複数の世界観を載せるフレームワーク」として設計する。

そのため、開発上の優先順位は以下とする。

1. メインループを壊れない形で定義する
2. 共通機能名を先に固定する
3. 世界観差分は表示と演出に寄せる
4. 必要最小限の世界観パック仕様を定める
5. 追加や生成が容易な構造を保つ
